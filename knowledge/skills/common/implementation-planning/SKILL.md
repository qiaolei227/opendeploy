# 实施 plan md 模板与节奏

## 什么时候要出 plan(而不是直接 design)

- **一步到位的小改动**(加一个字段校验、调一个联动) → 不要 plan,直接 design + 签字 + 落地(base-system 硬规则一)
- **多步 / 跨层组合** → 要 plan
- **多个独立子项**(用户一次丢来好几件事) → 先分解,然后**每个子项一份自己的 plan**,不要往一份 plan 里塞多个独立需求

判断信号:
- 同一个需求既要"启用标准模块"又要"改 BOS 配置"又要"写插件" → 多步组合,plan
- 需求里有多个独立动词("加 X,同时 Y,再做个 Z") → 多子项,分解
- 步骤里有依赖顺序(A 完成 B 才能做) → plan 能把依赖画清楚

## Plan md 的位置

```
~/.opendeploy/projects/<projectId>/plans/YYYY-MM-DD-<短主题>.md
```

**文件名用具体主题**(`credit-management-upgrade.md` 而非 `plan-1.md`)——这些 md 本身就是**交付档案**,客户以后要翻的。

## 模板

```markdown
# <一句话概括做什么>

**项目**:<项目名>(数据库 <database>)
**负责人**:<顾问姓名 / 不知道就留空让用户填>
**日期**:YYYY-MM-DD

## 目标

<2-4 句话:客户想解决什么业务问题、这次要达到什么可观察到的效果>

## 验收标准

具体、可验证的几条(顾问交付时逐条勾):

- [ ] <标准 1 · 动作可复现、结果可观测>
- [ ] <标准 2>
- [ ] ...

## 步骤

### 步骤 1 · <[标准启用] / [BOS 配置] / [Python] / [交付]> <简述>

- **Owner**: 你 / 我
- **动作**:
  1. ...
  2. ...
- **验证**:
  - ...
- **备注**:(可选,风险点 / 前置条件)

### 步骤 2 · ...

...

## 已知风险

- <风险 1 → 缓解 / 回滚方式>
- <风险 2 → ...>

## 回滚路径

<如果这一整份 plan 全部撤掉,按什么顺序 / 用什么工具 / 依赖哪些 backup 文件>
```

## 字段约定

### 步骤层级标签(必须打)

每个步骤的标题带一个标签,**明确是哪一层的动作**:

| 标签 | 含义 | 典型 owner |
|---|---|---|
| `[标准启用]` | K/3 Cloud 标准模块开关 / 系统参数 | 你(顾问在界面点) |
| `[BOS 配置]` | BOS Designer 里的无代码配置(字段 / 规则 / 审批流 / 套打) | 你(agent 不自动化) |
| `[Python]` | Python 插件 | 我(agent 调工具) |
| `[DLL]` | .NET DLL 插件 | 外部二开(v0.1 不支持) |
| `[数据]` | 批量导入 / 初始化基础资料 | 你 或 我(视工具覆盖) |
| `[交付]` | F5 刷新、客户端重登、SVN commit、用户培训 | 你 |

### Owner 字段

只两种值:

- **你** = 用户(顾问 / 客户)要亲手做的
- **我** = OpenDeploy agent 通过工具自动做的

**不要**用"顾问 A / 客户 IT"这种人名——plan 跨人流转,人名会过时;角色(你 / 我)永远清楚。

### 动作与验证分开写

- **动作** = 要做的具体操作(点哪个菜单 / 调哪个工具 / 传什么参数)
- **验证** = 怎么确认这一步做成了(看哪里的状态变了 / 调哪个只读工具反查)

**验证不能省**——base-system 硬规则四要求每一步都要有验证。

## 执行节奏

1. **plan 写完先给用户签字**。说"这是我准备做的步骤清单,请 review——每一步的层级、owner、验收标准你都同意吗?有没有漏项或多余?"用户说"OK"再开始。
2. **逐步执行,每步 checkpoint**。每完成一步(尤其是 owner = 我 的步骤):
   - 报告:刚做了什么、验证结果是什么
   - 问用户:"下一步是 [步骤 N+1 · xxx],owner 是 [你 / 我],确认继续?"
   - 用户说 OK 再继续。**不要连环跑完**。
3. **用户说"全部执行" / "一路走到底"才能不停**——这是用户主动放弃 checkpoint,可以一口气跑,但**每一步仍然要验证**,失败就停。
4. **中途出现 plan 外的事**(工具报错、用户插话改需求),停下来,**不要偷偷调整 plan**。回到 design / 澄清 / 排错流程。

## 抽象示例骨架

按下面的形态写自己的 plan(占位符 `<...>` 替换为本次需求的真实内容):

```markdown
# <一句话点透做什么>(<可选副标题:具体到的子目标>)

**项目**:<项目名>(数据库 <database>)
**日期**:YYYY-MM-DD

## 目标

<2-4 句话,讲清当前痛点 + 这次要达到什么效果>

## 验收标准

- [ ] <可观察、可勾选的标准 1>
- [ ] <标准 2>
...

## 步骤

### 步骤 1 · [<层级标签>] <一句话>

- **Owner**: 你 / 我
- **动作**: 1. ... 2. ...
- **验证**: <反查方式>

### 步骤 2 · [<层级标签>] ...
...

## 已知风险

- <风险> → <缓解>

## 回滚路径

逐步对应,从最后一步往前撤。
```

**完整真实示例**(4 步组合,信用管控升级):看 `examples/credit-management.md` 文件——里面是把上面骨架完全填实的样本,每一段都按真实金蝶字段名 / 工具名写。新需求按那份的丰满度写自己的 plan。

通过 `load_skill_file('common/implementation-planning', 'examples/credit-management')` 加载完整示例。

## 反模式

| ❌ | ✅ |
|---|---|
| 一口气把 plan 4 步跑完才报告 | 每步 checkpoint,用户确认 |
| Owner 写"客户 IT 部" / "我们这边" | 只用**你 / 我** |
| 步骤里只有"修改 xxx",没写具体动作 | 动作 + 验证分写,每步都能事后核查 |
| 没有回滚路径 | 每份 plan 都得有"全撤回怎么走" |
| 中途需求变了直接改 plan md 就继续 | 停下,回澄清,必要时重写 plan |
| 一份 plan 塞 5 个独立需求 | 拆成 5 份 plan |
