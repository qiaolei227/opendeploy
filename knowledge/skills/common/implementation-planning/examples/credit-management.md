# 示例 plan:信用管控升级

> 这是 `implementation-planning` 的**完整真实场景示例**,展示一个 4 步组合需求的 plan md 长什么样。SKILL.md body 里只放抽象骨架,真实需求时按这个示例的丰满度写。
>
> 场景虚构,字段名是真实的 K/3 Cloud 字段。

---

```markdown
# 信用管控升级(超额预警 + 黑名单自动拦截)

**项目**:川沙诚信商贸(数据库 AIS20260302144343)
**日期**:2026-04-25

## 目标

当前客户只用人工在销售订单上看客户欠款。需要:
1. 启用系统内置信用管理模块,自动核算客户可用额度
2. 特殊标记的"黑名单客户"在保存销售订单时直接阻断
3. 普通客户超限时给软提示

## 验收标准

- [ ] 销售订单保存前自动核算信用占用,显示当前额度 / 占用
- [ ] 黑名单客户(客户档案标记 `F_IsBlacklisted=true`)保存时阻断并提示
- [ ] 非黑名单客户超额只弹提示,可继续保存
- [ ] 配套字段 / 插件都有 backup 文件,可一键回滚

## 步骤

### 步骤 1 · [标准启用] 开启信用管理模块

- **Owner**: 你
- **动作**:
  1. 进"系统服务云" → "信用管理" → 启用
  2. 给所有现有客户档案初始化信用额度(可批量导入)
- **验证**: 销售订单界面能看到"可用信用额度"字段

### 步骤 2 · [BOS 配置] 客户档案加"黑名单"标记字段

- **Owner**: 你(OpenDeploy v0.1 不自动化扩展字段)
- **动作**:
  1. BOS Designer 打开 BD_Customer 扩展
  2. 新增字段 `F_IsBlacklisted`,类型复选框
- **验证**: 客户档案列表 / 表单都能看到该字段,能编辑保存

### 步骤 3 · [Python] 销售订单 BeforeSave 校验

- **Owner**: 我
- **动作**:
  1. `kingdee_list_extensions('SAL_SaleOrder')` 看现有扩展
  2. 生成 pyBody,挂 `BeforeSave` 拦截,读 `F_IsBlacklisted` 决定 hard block / soft warn
  3. 调 `kingdee_register_python_plugin` 或 `kingdee_create_extension_with_python_plugin`
- **验证**:
  - `kingdee_list_form_plugins` 反查插件落库
  - 建一个测试黑名单客户,手工试保存一单,应阻断
  - 建一个普通客户超额,应只提示可继续

### 步骤 4 · [交付] 刷新 + 培训

- **Owner**: 你
- **动作**:
  1. BOS Designer 按 F5 刷新
  2. 所有在线客户端重登
  3. SVN 同步(如团队协作)
  4. 给销售部同事说明黑名单标记的启用 / 撤销流程
- **验证**: 目视确认

## 已知风险

- 黑名单阻断误伤历史订单的补录场景 → 告知销售"补录历史单请临时把 F_IsBlacklisted 改 false,录完恢复"
- 信用管理启用前的存量订单额度未重算 → 后台 job 跑一次或接受现状

## 回滚路径

1. 步骤 3 的 backup:`<backup 路径>`,用 `kingdee_restore_from_backup` 还原
2. 步骤 2 的字段:BOS Designer 手工移除(OpenDeploy 不自动化)
3. 步骤 1 的信用管理:系统参数关闭即可
```

---

## 这份示例展示了什么(写新 plan 时照搬这些做法)

1. **标题一句话点透**——"信用管控升级(超额预警 + 黑名单自动拦截)" 比 "信用管理需求" 信息量大十倍
2. **目标段三句话讲清业务问题** + **验收标准 4 条都可勾选**
3. **每步打 `[层级标签]`**,让客户也能一眼看出哪些是配置 / 哪些要写代码
4. **Owner 只用 你 / 我**,不用人名
5. **步骤 3 的"动作"和"验证"分开**,验证用反查工具具体到位
6. **回滚路径**逐步对应,每步都有撤回方式
