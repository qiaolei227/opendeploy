# 行业示例:跨境电商 实施速查 🟡

> 帮 OpenDeploy agent 在收到"我们是跨境电商客户"线索时形成行业上下文。**非单 plan**,是行业级速查——用来快速对齐客户画像、识别"K/3 Cloud 能做什么不能做什么"的边界,尤其是 OMS / WMS / 平台对接的系统分工。

---

## 1. 行业画像

- **典型客户规模**:卖家 SKU 几百到几万,年营收从千万到数十亿;店群 / 铺货卖家 / 品牌卖家 / 独立站卖家模式各异
- **业务模式**:多平台铺货(Amazon / Shopify / eBay / Temu / TikTok Shop / 国内电商 / 独立站)+ 多币别结算 + 海外仓 / FBA / 国内仓混用 + 报关出口退税 + 跨境结汇
- **K/3 Cloud 适配度**:🟡 中等。**星空擅长做跨境企业的"业财一体化后台"——财务 / 供应链 / 核算**;**订单前台 / 多平台抓单 / 物流跟踪通常由跨境电商 ERP(领星 / 积加 / 易仓 / 旺店通等)承担**,通过 WebAPI / iPaaS 和星空对接
- **常见版次**:V9 旗舰版(多币别 / 多准则 / 多法人 / 税务云通常都要),企业版在多组织 + 跨境结算上够用但旗舰更齐全
- **国际化定位**:金蝶云·星空旗舰版支持**多语言 / 多币种 / 多时区 / 多会计准则**,对跨境集团化管控友好
- **行业子细分(实施侧重不同)**:
  - 亚马逊精品卖家 → 重 FBA 库存 / 广告费分摊 / 平台费用核算
  - 铺货 / 店群卖家 → 重多平台 + 多店铺 + 多账号管控,K/3 Cloud 通常合并报表层
  - 独立站品牌出海 → 重多主体合并 + 多币别 + 海外公司账务
  - 跨境 B2B / 外贸 → 重报关 / 退税 / 信用证 / 国际结算,接近传统外贸 ERP 实施
  - 工厂型出海 → 生产 + 跨境一体,启用 MFG + 跨境模块组合

**关键判断**:客户说"跨境电商" → **先问"你们 K/3 Cloud 在整体架构里定位"**。99% 的场景是:
1. **K/3 Cloud = 后台 ERP + 财务中台**(主流):跨境 ERP 做订单 / 物流 / 平台对账,业财数据传到星空出凭证 / 退税 / 合并报表
2. **K/3 Cloud 直接对接平台**(少见):个别精品卖家单平台直连,但性价比低,维护成本高
3. **多主体 / 多法人**(品牌出海 + 独立站集团):国内公司 + 海外子公司 + 香港离岸公司,用星空旗舰版跨组织 + 多准则

## 2. 典型业务场景与痛点

(客户原话风格;每条建议 agent 收到这样的描述时**先不要立刻设计方案,问清场景背景**)

1. **"Amazon / Shopify / 独立站订单能自动进 ERP 吗"** → 不直连,走跨境 ERP 或 iPaaS;问清他们用的是哪家(领星 / 积加 / 易仓 / 旺店通 等)
2. **"每个平台币种不一样,财务怎么入账"** → 多币别 + 期末调汇,标准能力;[应收期末调汇](https://help.open.kingdee.com/dokuwiki/doku.php?id=%E5%BA%94%E6%94%B6%E6%9C%9F%E6%9C%AB%E8%B0%83%E6%B1%87) 是启用项
3. **"海外仓 + FBA + 国内仓怎么统一管"** → 多仓库 + 多库存组织,库存状态扩展("在途 / 待检 / 冻结")
4. **"FBA 库存和平台后台对不上"** → 通常是跨境 ERP 对接问题,不是星空问题;推客户查跨境 ERP 的 FBA 同步日志
5. **"出口退税单证怎么和 ERP 串起来"** → 星空旗舰版**内置出口退税模块**(生产企业 / 外贸企业);非旗舰版通常外挂税务云 / 报关软件
6. **"多个平台结算周期不一样,应收对账太乱"** → 按平台拆客户档案 + 按币别 + 自定义账龄,**不需要二开**
7. **"集团下有国内公司 / 香港公司 / 美国公司,报表怎么合并"** → 多准则 + 合并报表(旗舰版能力);实施复杂,通常需要集团财务团队深度参与
8. **"平台订单号 / 物流单号在 ERP 里要能反查"** → BOS 扩展字段(销售订单 / 销售出库加平台单号字段),不是 Python
9. **"每个 SKU 在不同平台 listing title 不一样"** → 通常跨境 ERP 管 listing,K/3 Cloud 只管中性物料编码 + 内部名称;**不要**在星空里塞多语言 title
10. **"亚马逊扣了一笔费用想分摊到订单上"** → 平台扣费 / 广告费 / 仓储费,通常是跨境 ERP 先计算分摊后再推总账,K/3 Cloud 只收凭证
11. **"跨境 ERP 推过来的凭证老是错的"** → 多数是数据映射问题(币种 / 科目 / 客户主体),实施时要重点走通 3-5 个典型业务场景再全量上
12. **"VAT / GST 申报怎么配合"** → 不同国家税制不同,**通常不在 K/3 Cloud 内申报**,走本地税务代理;星空只记应交税金

## 3. 标准模块组合

跨境电商典型启用顺序(**重财务 + 重多币别 + 重集成**):

| 模块 | 用途 | 启用顺序 |
|---|---|---|
| BD(基础资料) | 多语言物料 / 多币别客户 / 多主体供应商 | 第 1 批(必做) |
| 财务会计(FIN)+ AR / AP / CB | 凭证 / 应收应付 / 资金;**多币别必启** | 第 1 批(必做) |
| 多组织 + 多币别 + 多准则 | 集团 + 子公司 + 离岸公司 | 第 1 批(集团型必做) |
| 供应链:PUR / STK / SAL | 采购入库 / 库存 / 销售出库 | 第 1 批(必做) |
| 存货核算 | FIFO / 加权平均;跨境常用 FIFO 配合批次 | 第 1 批(必做) |
| 多仓库(国内仓 / 海外仓 / FBA 仓虚仓)| 库存组织 + 仓库多层管控 | 第 1 批 |
| 税务云 / 出口退税(旗舰版)| 出口报关 + 免抵退税申报 | 第 2 批(按业务类型) |
| 合并报表 | 集团多主体合并 | 第 2 批(集团型) |
| WebAPI / iPaaS 集成 | 跨境 ERP / 电商平台 / 物流商 | 贯穿 |

**重点**:跨境 ERP(领星 / 积加 / 易仓 / 管易云)通常是**平台订单抓取 + 物流追踪 + 平台财务对账**的主系统,K/3 Cloud 是**后端业财**。两者分工明确,实施前必须画清边界图。<!-- 待实证:具体哪些跨境 ERP 有现成星空对接包,以及各自的同步粒度 -->

## 4. 高频自定义层级

跨境电商需求**超过一半是"集成边界 + 多币别配置 + 多组织核算"问题**,不是二开。按"标准 > BOS > Python > DLL"四层判断:

### 4.1 标准能搞定(先反查)

- 多币别核算 + 期末调汇 → 启用多币别 + 汇率表 + 期末调汇向导
- 多主体财务合并 → 多准则 + 合并报表
- 出口退税台账 → 旗舰版税务云标准
- 海外仓 / 国内仓分别管 → 多库存组织 + 多仓库
- 多仓库存实时查询 → 即时库存 N 维汇总
- 多账簿记账(内账 / 外账 / 管理账)→ 多会计准则

### 4.2 BOS 配置层(无代码)

- 销售订单 / 销售出库加平台单号、物流单号、平台扣费字段 → BOS 字段扩展
- 客户档案加平台 / 店铺 / 站点维度 → BOS 扩展 BD_Customer
- 销售订单按平台 / 站点走不同审批 → 审批流 + 条件
- 自定义集团报表视图 → BOS 配置视图

### 4.3 Python 插件(agent 可自动化)

- 销售订单保存时按平台规则拆单 / 校验 → BeforeSave
- 多币别转换本位币的自定义舍入逻辑(客户自己规则)→ BeforeSave
- 跨境 ERP 推过来的数据校验 / 字段联动 → BeforeSave + DataChanged
- 海外仓库存状态自定义流转 → BeforeSave / AfterSave

### 4.4 DLL / 外部二开 / 客户另选系统(v0.1 不支持或不该做)

- 多平台订单抓取(亚马逊 MWS / SP-API、Shopify、独立站 API)→ **跨境 ERP 的事,不在 K/3 Cloud 做**
- FBA 库存实时同步 → **同上**
- 物流商 API 对接(UPS / DHL / FedEx / 本地 carrier)→ **同上**
- VAT / GST 自动申报 → **本地税务软件的事**
- 汇率自动抓取(央行 / 平台牌价)→ **通常外挂脚本抓取后导入**

## 5. 实施常见坑 🟡

1. **边界画不清,把跨境 ERP 的活塞进星空** → 最常见坑。客户说"订单自动进 ERP",顾问一拍脑袋写 WebAPI + 大量插件;真相:客户已经有领星 / 积加等,标准跨境 ERP 有现成对接方案。**启动 Kickoff 先画数据流图**——平台 / 跨境 ERP / K/3 Cloud / 银行 / 报关 各自职责。
2. **多币别没启动就录了业务单** → **启用多币别是初始化阶段的动作**,后期补启用非常痛苦。实施第一周就要定币别清单 + 汇率机制。
3. **汇率机制没和财务确认** → 用即时汇率 / 月平均 / 期初固定 / 交易日汇率?不同客户财务政策不同,**必须先问财务再配**。
4. **海外仓 FBA 被当成"普通仓库"建** → FBA 仓是亚马逊管的,**K/3 Cloud 里的 FBA 仓是镜像**,和真实库存会有时差;实施时要明确"镜像仓"规则和对账频率。
5. **物料编码体系没统一** → 国内品 / 海外品 / 不同平台 listing 编码不同 → 跨境 ERP 推数据时映射地狱。**物料档案总部集中管控,跨平台 listing 在跨境 ERP 里维护映射**。
6. **出口退税单证链路断** → 报关单 / 发票 / 收汇凭证 / 出口发票要串联到销售订单;旗舰版支持,但**配置链路长,需要财务 + 报关员 + IT 三方对齐**。
7. **集团合并报表试图一次上完** → 多主体合并是顶层需求,实施复杂度高;**先单主体跑通 3 个月,再上合并**。
8. **把跨境 ERP 当成"平台适配器"临时用** → 很多客户觉得"我就对接一个 Amazon,用跨境 ERP 大材小用" → 自己写 SP-API + 自建扣费分摊 → 3 个月后维护崩溃。**宁可多花钱用专业跨境 ERP**。

## 6. WebAPI / iPaaS 集成(跨境特重度场景)

跨境电商的集成**比连锁零售还重**,至少 **4-6 条通道**:

| 集成对象 | 典型方向 | OpenDeploy 支持 |
|---|---|---|
| 跨境 ERP(领星 / 积加 / 易仓 / 管易云)↔ K/3 Cloud | 订单 / 发货 / 财务凭证双向 | ❌ 不工具化,给客户 IT 指引 |
| 汇率数据源 → K/3 Cloud | 定期导入汇率表 | ❌ 同上 |
| 报关 / 退税软件 ↔ K/3 Cloud | 报关单 / 退税单证 | ❌ 同上 |
| 海外仓 WMS(易仓 / 万邑通 / 第三方海外仓系统)↔ K/3 Cloud | 出入库 / 库存余额 | ❌ 同上 |
| BI / 数据仓库 ← K/3 Cloud | 查询类端点推送业务数据 | ❌ 同上 |
| 独立站 / Shopify 直连(少见)→ K/3 Cloud | 订单直推 | ❌ 同上 |

**详细指引加载** `load_skill_file('k3cloud/webapi-index', 'references/integration-patterns')`(当 WebAPI skill 可见时)。

**OMS / WMS 边界(务必和客户对齐)**:
- **K/3 Cloud 不是 OMS(订单管理系统)** —— 平台订单抓取 + 多平台订单归一 + 路由决策应由专业 OMS / 跨境 ERP 承担
- **K/3 Cloud 不是 WMS(仓储管理系统)** —— 海外仓的波次拣货 / 上架策略 / 库位优化应由专业 WMS 承担;K/3 Cloud 做"库存账面 + 财务核算"
- 客户说"我不想上那么多系统" → 先问清业务量级;月订单量 < 5000 单可以尝试用星空 + BOS 扩展凑合;超过这个量级坚持不上 OMS / WMS,**通常实施 6 个月后推倒重来**

## 7. 需求快速分诊表

收到客户描述时,先按这张表分诊,**别一上来就想"做什么"**,先想"该谁做":

| 客户原话 | 第一反应不是 | 真实路径 |
|---|---|---|
| "Amazon 订单自动进 ERP" | 不是写 SP-API 调用 | 问跨境 ERP(领星 / 积加 / 易仓 / 旺店通)选型,通常已有现成对接 |
| "FBA 库存实时看" | 不是同步 FBA 数据到星空 | 跨境 ERP 看 FBA 库存;星空只看"账面 + 库存核算"层 |
| "多平台扣费分摊到订单" | 不是写分摊算法插件 | 跨境 ERP 端分摊后推总账;星空 BOS 加字段接收即可 |
| "VAT / GST 自动申报" | 不是在 K/3 Cloud 内做 | 各国本地税务软件 / 代理;星空只记应交税金 |
| "汇率每天自动抓" | 不是写抓取脚本 | 客户脚本 / iPaaS / 手工导入;星空汇率表标准 |
| "出口报关单和销售订单挂起来" | 不是开发字段联动 | 旗舰版税务云 + 出口退税模块标准支持 |
| "国内公司 + 香港公司报表合并" | 不是手工合并 | 多准则 + 多组织 + 合并报表(旗舰版) |
| "海外仓和国内仓在同一张表里看" | 不是新报表 | 即时库存查询多维过滤 |
| "跨境 ERP 推过来的订单老对不上" | 不是改星空 | 查跨境 ERP 端的映射 + 主数据一致性 |
| "想让星空成为 OMS / WMS" | **强烈劝阻** | 月单量 > 5000 必须上专业 OMS / WMS;星空做不了订单中台 |

## 8. 实施节奏建议(典型 12-24 周)

跨境项目比连锁零售复杂度更高,周期也更长。

| 阶段 | 周次 | 交付 |
|---|---|---|
| 蓝图 / 边界画图 | W1-W3 | 系统架构图(平台 / 跨境 ERP / K/3 Cloud / 银行 / 报关 各自职责)+ 数据流图 + 主数据归属确认 |
| 主数据 + 多币别启用 | W4-W6 | 物料 / 客户 / 多组织 / 多币别 / 汇率机制 / 期初导入 |
| 业务流程串联(国内主体)| W7-W11 | 采购 → 库存 → 销售 → 出库 单主体跑通 |
| 集成对接 | W8-W14(并行) | 跨境 ERP / 海外仓 WMS / 报关软件 等接入,逐场景 POC |
| 跨主体 + 多币别核算 | W12-W16 | 多组织间结算 + 期末调汇 + 多准则 + 海外子公司账务 |
| 出口退税(如需)| W14-W18 | 启用税务云退税模块 + 单证链路 + 试报税 |
| 合并报表(集团型)| W17-W20 | 单主体跑 3 个月稳定后才上;**别一上来就做合并** |
| UAT + 培训 | W19-W22 | 用户验收 + SOP 文档 + 财务团队培训 |
| 上线 + 跟进 | W23-W24+ | 切系统 + 至少跑通 1 个完整月结 |

**实施红线**:
- 跨境 ERP 厂商 / 报关软件供应商 / 银行结算渠道**必须 W1 全员进项目组**,边界画不清就开干是无底洞
- 多币别 / 多组织 / 多准则**必须初始化阶段就定**,后期补启用代价巨大
- 合并报表是**最后做的**,不是先做的;单主体没跑稳定就上合并 = 自找麻烦
- 出口退税涉及税务合规,**必须有税务顾问 / 财务总监深度参与**,顾问别自己拍板

## 9. 示范 plan 链接

跨境电商典型 plan 雏形(本目录暂未落盘,可参考 `credit-management.md` 格式自行撰写):

- **跨境 ERP 接入 K/3 Cloud 业财分离方案**:画数据流图 → 定主数据归属 → 启用多币别 → 配置 WebAPI 接入点 → 3 个典型场景 POC → 全量
- **多主体集团合并初始化**:多组织启用 → 多准则映射 → 核算政策对齐 → 单主体跑 3 个月 → 合并报表试算
- **海外仓库存状态管理**:库存组织拆 → 库存状态扩展(在途 / 待检 / FBA 冻结)→ BOS 加仓库属性字段 → Python 校验入库单合规
- **出口退税单证链路搭建**:启用出口退税模块 → 主数据配置 → 报关单 ↔ 销售订单映射 → 退税申报试跑(**需要税务顾问深度参与**)
- **平台费用 / 广告费分摊到订单**:跨境 ERP 推数据时分摊算法在跨境 ERP 端 → K/3 Cloud BOS 加平台费分摊字段 → Python 接收时校验
- **汇率定期导入与期末调汇 SOP**:汇率表配置 → 月初汇率导入(脚本或手工)→ 月末调汇向导跑通 → 财务团队接手培训

每个 plan 都遵循 implementation-planning SKILL.md 的"标 + 步 + 验 + 滚"模板。**跨境项目尤其要 plan 在前——业务复杂度高,边界画错就是无底洞**。

## 临时话术

如果客户需求在以上场景都没完全对得上,agent 这么说:

> "跨境电商我整理的是行业画像 + 系统分工边界 + 集成模式,具体到您这家的实施方案,我需要先问四件事才能给出靠谱判断:
>
> 1. **整体系统架构** —— 你们有没有用跨境 ERP(领星 / 积加 / 易仓 / 旺店通 / 管易云)?K/3 Cloud 定位后台还是前台?
> 2. **业务主体和币别** —— 几个法人?涉及几个币种?汇率机制是什么(即时 / 月平均 / 交易日)?
> 3. **海外仓策略** —— 自建 + FBA + 第三方海外仓的组合?月订单量大概多少?有没有现成 WMS?
> 4. **退税 / 报关 / 合规** —— 是否走退税?国外税务怎么处理(自办 / 外包代理)?
>
> 这四条明确后,跨境电商的实施思路通常非常清晰:**K/3 Cloud 做业财后台,其他交给专业系统,通过 WebAPI / iPaaS 串起来**。真正需要 Python 插件的场景不超过 3-5 个,而且都在'数据接收时的校验和分摊'这一层。"
