# 行业示例:连锁零售 实施速查 🟡

> 帮 OpenDeploy agent 在收到"我们是连锁零售客户"线索时形成行业上下文。**非单 plan**,是行业级速查——用来快速对齐客户画像、识别标准能力边界、提前警惕行业特有坑。

---

## 1. 行业画像

- **典型客户规模**:总部 + 10-500 家门店;常见"直营 + 加盟 + 经销"混合模式;自营电商(旗舰店 / 官网)通常也并入同一套 ERP 财务
- **业务模式**:总部集采 / 总仓分仓 / 门店零售 / 会员营销 / 日结月结;订单量高、单价低、流水密集
- **K/3 Cloud 适配度**:🟡 中等。**星空擅长供应链 + 财务的后台 ERP**;**前端 POS / 会员中台 / 全渠道营销常走独立系统**(金蝶星辰零售云 / 第三方 POS),通过 WebAPI 或专用电商集成方案回传订单
- **常见版次**:V9 标准版 / 企业版(多门店 + 多组织协同通常上企业版)
- **行业子细分(实施侧重不同)**:
  - 服饰 / 鞋包 → SKU 多 + 季节性 + 退换频繁,重 SKU 维度报表
  - 食品 / 生鲜 → 保质期 + 批号 + 临期处理,重 STK 启用项
  - 餐饮连锁 → 中央厨房 / 加工 / 派送,重生产 + 配送
  - 化妆品 / 母婴 → 重会员 + 重营销规则,K/3 Cloud 后台为辅
  - 家居 / 家电 → 大件物流 + 预约配送,重订单履约

**关键判断**:客户说"连锁零售" → **先问清"K/3 Cloud 在客户架构里扮演什么角色"**。常见三种:
1. **只做后台财务 + 供应链**(主流):POS / 会员走独立系统,K/3 Cloud 收订单 / 跑库存 / 出应收凭证
2. **全套上星空**(罕见):门店端也跑 K/3 Cloud 客户端或 Web 前台——实施复杂度高
3. **加盟商管理**:总部用 K/3 Cloud,加盟商各自系统,通过渠道云 / 对账单交互

---

## 2. 典型业务场景与痛点

(客户原话风格;每条建议 agent 收到这样的描述时**先不要立刻设计方案,问清场景背景**)

1. **"每家店的库存要独立看,但总部要能一眼看全盘"** → 通常指多组织库存 + 合并查询,不是新报表需求
2. **"门店急需要货,要从别的门店调"** → 直接调拨单 / 分步式调拨(在途库存)
3. **"POS 卖了一单,库存什么时候扣"** → 实施时确认 POS 和 K/3 Cloud 的对接粒度(实时 / T+1 / 手工日结)
4. **"会员卡储值能不能在星空里管"** → 🟡 星辰零售云支持,星空主线标准能力较弱,通常外挂零售云或第三方会员系统
5. **"满 200 减 30、全场 8 折、买一送一能不能配"** → 参考 [促销分析](https://help.open.kingdee.com/dokuwiki_std/doku.php?id=%E4%BF%83%E9%94%80%E5%88%86%E6%9E%90) 功能;**复杂促销规则通常由前端 POS / 电商中台承担**,K/3 Cloud 只收最终订单金额
6. **"日结对账每天晚上折腾几个小时"** → 通常是 POS 流水和销售订单的核对流程,先查客户现有流程瓶颈在哪(数据量 / 差异定位 / 凭证生成),不是一下子就做插件
7. **"总部要看各店排行榜"** → 报表类需求;应收 / 销售分析有内置,复杂场景走 BI 或自定义报表
8. **"月结关账前总有一堆未审核的 POS 日结单据"** → **库存关账前必须处理掉**(参考 stk-inventory 关账检查)—— 流程优化 + 关账前检查,不是二开
9. **"新开一家店要建档案快一点"** → 基础资料批量导入,或 BOS 建档案模板
10. **"加盟商的销售数据要拿回来做对账"** → 渠道云 / 对账单 + WebAPI 回传,属集成项,不是单机二开

## 3. 标准模块组合

零售连锁典型启用顺序(**先后台,再前台,最后营销层**):

| 模块 | 用途 | 启用顺序 |
|---|---|---|
| BD(基础资料) | 组织架构 / 门店档案 / 物料 / 客户 / 供应商 | 第 1 批(必做) |
| 供应链:PUR / STK / SAL | 采购入库 / 库存 / 销售出库(门店销售通常在 SAL 或独立零售单据) | 第 1 批(必做) |
| 存货核算 | 计价方法 / 库存金额 / 月结 | 第 1 批(必做) |
| 财务会计(FIN)+ AR / AP / CB | 凭证 / 应收应付 / 银行资金 | 第 1 批(必做) |
| 多组织 | 总部 / 区域 / 门店三层组织;库存组织独立核算 | 第 2 批(连锁必做) |
| 直接调拨 / 分步式调拨 | 总仓 → 门店 / 门店间调货 | 第 2 批 |
| 寄售业务 | 加盟商代销 / 铺货再结算 | 第 2 批(按模式) |
| 全渠道云 / 零售云 / 渠道云 | 前端 POS / 会员 / 电商订单中台 | 第 3 批(按客户架构) |
| WebAPI 集成 | 回写 POS 订单 / 同步会员数据 / 接第三方电商 | 贯穿 |

**注意**:K/3 Cloud 本身**不自带 POS 收银前台**——需要第三方 POS(客户常用管家婆 / 银豹 / 美团收银等)或金蝶星辰零售云。实施启动前要和客户确认 POS 选型。<!-- 待实证:具体哪些 POS 有现成对接包 -->

## 4. 高频自定义层级

连锁零售的需求**超过 70% 是标准 + 配置能覆盖的**。按"标准 > BOS > Python > DLL"四层判断:

### 4.1 标准能搞定(先反查)

- 多组织库存隔离 + 汇总查询 → 多组织 + 即时库存查询
- 跨门店调货 → 直接调拨 / 分步式调拨
- 日结月结 → 库存关账 + 存货核算月结
- 销售排名 / 利润分析 → 内置销售分析报表 / 利润表
- 加盟商代销 → 寄售类客户
- 基本满减 / 折扣 → SAL 价目表 + 价格策略

### 4.2 BOS 配置层(无代码)

- 门店档案加自定义字段(店长 / 店型 / 开业日期) → BOS 扩展 BD_Customer 或自建基础资料
- 销售订单按门店类型走不同审批 → 审批流 + 条件表达式
- 自定义套打(小票 / 对账单 / 发货单) → 打印模板
- **加门店维度的分析视图** → 即时库存查询 + 过滤方案(不需新报表)

### 4.3 Python 插件(agent 可自动化)

- 销售出库保存校验(总部价格 vs 门店价格一致性)→ BeforeSave 事件
- 门店日结汇总表生成触发(累计某些字段) → AfterSave 或自定义 Button
- POS 回传订单时的字段联动校验 → BeforeSave
- 特殊促销规则(标准促销配不出来的组合)→ **先确认标准和 BOS 都搞不定再走 Python**

### 4.4 DLL / 外部二开(v0.1 不支持)

- 和 POS 的深度实时对接(毫秒级库存锁定)
- 会员积分复杂计算引擎(跨等级跨品类)
- 自研报表引擎嵌入

## 5. 实施常见坑 🟡

1. **POS 对接粒度没提前对齐** → 实施后期才发现 POS 是 T+1 导入,客户要"实时库存",全部返工。**启动 Kickoff 就要定对接时效协议**(实时 / 准实时 / T+1 / 手工日结),明确异常场景(网络断开 / 平台限流)的降级策略。
2. **门店库存组织设得太细** → 把每家门店都设成独立库存组织,结果跨店调货得走跨组织结算,流程复杂;实际多数连锁用**"总仓组织 + 门店仓(仓库级)"**就够。库存组织设置前**必须和客户确认财务核算粒度**(是不是每个店要独立出报表 / 是不是每个店独立法人)。
3. **促销规则塞进 K/3 Cloud** → 复杂促销(满减叠加 / 会员折 / 赠品组合)应该在 POS / 中台处理,K/3 Cloud 只收最终金额 + 折扣原因字段。硬塞进来通常得写大量插件且维护成本高;**促销规则变更频繁的零售场景尤其不要硬塞**。
4. **多组织同物料编码不统一** → 总部 / 门店 / 加盟商各自建了物料档案 → 调拨 / 对账全乱。**物料档案必须总部集中管控,门店只能选不能建**;基础资料权限 + 数据规则一定要前置配置好。
5. **日结 / 月结前未审核单据堆积** → 关账失败,被迫加班补。**系统参数 + 审核流程规范** 是主因,不是二开;通常是日结 SOP 缺失或没人盯异常单据。
6. **多组织价目表混乱** → 不同门店不同价,价目表没按组织拆 → BOS 级字段扩展通常解决不了,**价目表 + 价格组正确配置**是标准解法。
7. **会员 / 储值卡硬塞星空** → 星空会员管理能力弱,**推客户走零售云 / 第三方会员系统 + WebAPI 回传**,别在星空里重建会员中台。
8. **门店开关业务时间窗** → 闭店期间 POS 没流水,但补录退货 / 调账常见;关账规则要考虑这种场景,**别配死禁止录入历史日期**。

## 6. WebAPI 集成(连锁零售特有重度场景)

连锁零售通常有至少 **3 条集成路径**:

| 集成对象 | 典型端点 | OpenDeploy 支持 |
|---|---|---|
| POS → K/3 Cloud(回传销售出库 / 收款)| `Sal_OutStock` 新增 + 审核 / `AR_Receivable` | ❌ 不工具化,给客户 IT 指引 |
| 电商平台(淘宝 / 京东 / 自营商城)→ K/3 Cloud(订单 / 发货 / 对账)| 管易云是金蝶子品牌,通常走专用集成方案 | ❌ 同上 |
| K/3 Cloud → BI / 大屏(销售数据推送)| 查询类端点 + 视图 | ❌ 同上 |

**详细指引加载** `load_skill_file('k3cloud/webapi-index', 'references/integration-patterns')`(当 WebAPI skill 可见时)。

**边界提醒**:OpenDeploy v0.1 **不自动化 WebAPI 调用**。agent 的角色是"告诉客户集成应该用哪个端点 / 怎么认证 / 哪些坑",实际代码由客户开发团队或第三方 iPaaS(轻易云 / 数环通 / 数帝等)完成。

## 7. 需求快速分诊表

收到客户描述时,先按这张表分诊,**别上来就想"做什么"**,先想"该谁做":

| 客户原话 | 第一反应不是 | 真实路径 |
|---|---|---|
| "门店 POS 卖一单 ERP 立刻知道" | 不是写实时同步插件 | 问 POS 选型 + 现有对接方案,通常是 POS 厂商接口或专用电商集成方案 |
| "总部要看每个店每天卖了多少" | 不是建新报表 | 销售分析报表 + 过滤方案 + 仪表盘 |
| "门店之间互相调货走审批" | 不是改调拨单 | 分步式调拨(标准带审批)+ 审批流配置 |
| "VIP 客户要单独留货" | 不是 BOS 加字段 | 销售订单锁库 + 预留网络 |
| "促销规则太复杂配不出" | 不是上来写 Python | 先确认是 K/3 Cloud 出促销还是 POS 出促销;90% 是 POS 出 |
| "新店开业要快速建档案" | 不是开发批量工具 | 基础资料导入向导 + 数据治理规范 |
| "加盟商每月对账太慢" | 不是写对账插件 | 寄售流程 + 渠道云 + 标准对账单 |
| "总部库存和门店库存对不上" | 不是查代码 | 查 POS 同步日志 + 关账前未审核单据 + 物料编码统一性 |
| "想要每个店一个独立账套" | **强烈劝阻** | 多组织(同一账套内)+ 财务隔离配置;独立账套合并报表非常痛苦 |

## 8. 实施节奏建议(典型 8-16 周)

| 阶段 | 周次 | 交付 |
|---|---|---|
| 蓝图 / Kickoff | W1-W2 | 系统架构图 + 边界协议 + POS 对接方式确认 |
| 主数据 + 基础启用 | W3-W5 | 物料 / 客户 / 门店 / 仓库基础资料 + 多组织启用 + 期初导入 |
| 业务流程串联 | W6-W9 | 采购 → 入库 → 调拨 → 销售 → 出库 全链路跑通;BOS 字段扩展;审批流 |
| 集成对接 | W7-W11(并行) | POS / 电商平台 WebAPI 接入,逐场景 POC |
| 财务核算 | W10-W12 | 凭证模板 + 月结流程 + 应收应付配置 |
| UAT + 培训 | W13-W14 | 用户验收 + SOP 文档 + 培训 |
| 上线 + 跟进 | W15-W16 | 切系统 + 7 天驻场 + 月结跑通 |

**实施红线**:
- POS 对接 / 电商集成的第三方厂商**必须 W1 进项目组**,不要等到 W7 才接触
- 月结 + 关账流程**至少要在 UAT 前完整跑通一遍**,别等真上线第一个月才跑
- BOS 字段扩展和 Python 插件**集中在 W6-W11**,W12 之后只做 bug fix 不做新需求

## 9. 示范 plan 链接

连锁零售典型 plan 雏形(本目录暂未落盘,可参考 `credit-management.md` 格式自行撰写):

- **门店库存预警 + 自动补货建议**:启用库存预警 → BOS 加门店类型字段 → Python 插件周期性算缺货 → 生成采购申请草稿
- **加盟商代销流程落地**:基础资料寄售类客户 → 销售订单类型寄售 → 寄售结算单 → 财务凭证
- **多门店调拨审批流**:BOS 调拨单加"紧急程度"字段 → 审批流按门店级别分路 → Python 校验总部库存
- **POS 日结差异核对辅助**:Python 生成差异汇总 → BOS 加对账状态字段(不替代日结流程,只辅助定位)
- **门店档案标准化批量初始化**:基础资料模板 → 数据导入向导 → BOS 加门店属性扩展字段(店型 / 商圈 / 开业日期)→ 上线后总部数据治理

每个 plan 都遵循 implementation-planning SKILL.md 的"标 + 步 + 验 + 滚"模板。**先 plan 后开干,不要看到需求就动手**。

## 临时话术

如果客户需求在以上场景都没完全对得上,agent 这么说:

> "连锁零售我整理的是行业画像 + 高频需求场景,具体到您这家的实施方案,我需要先问三件事才能给出靠谱判断:
>
> 1. **K/3 Cloud 在你们整体架构里定位什么** —— 纯后台 ERP / 全套上星空 / 加盟管理平台?这决定了实施边界
> 2. **POS 和会员系统用的什么** —— 有现成对接还是从 0 开始集成?需不需要走金蝶星辰零售云或渠道云?
> 3. **组织架构怎么设** —— 几层组织?哪层独立核算库存 / 财务?这是启动前必须对齐的
>
> 这三条明确后,80% 连锁零售的需求都是标准 + 配置 + 少量 BOS 就够,真正需要 Python 插件的场景不超过 2-3 个。"
