# 系统化排错纪律

## 什么时候进入排错模式

下列任一情况出现,**立刻**切换到排错模式(不要继续按原计划动工具):

- 工具返回 error
- 查询返回的结果和你预期的**完全不一致**(例如你用 `kingdee_get_fields` 查销售订单,返回空)
- 写入工具成功但**反查不到**(扩展 FID 在 `list_extensions` 里找不到、插件名在 `list_form_plugins` 里缺失)
- 用户反馈"按你说的做了,但没生效 / 报错了"
- 同一个操作**连续两次**表现出矛盾行为

## 反模式:猜测式重试

下列行为是**必须自觉纠正**的错误模式:

- "再试一次"(没形成假设的情况下)
- "换个字段名试试"
- "加个 `if` 判断绕过去"
- "把报错消息原样扔给大模型自己想"
- 不验证任何假设就直接调另一个工具试试

**每一次"再动手"之前,必须通过下面的五步流程产生证据**。

---

## 五步流程

### 1. 列症状 · 具体到可被事实核查

把问题描述成**可验证**的句子,而不是模糊感受。

| ❌ 模糊 | ✅ 具体可核查 |
|---|---|
| "插件不工作" | "`kingdee_register_python_plugin` 返回 success + backupFile,但 `kingdee_list_form_plugins('<extId>')` 返回空数组" |
| "查不到字段" | "对 `SAL_SaleOrder` 调 `kingdee_get_fields`,返回 0 条;但 BOS Designer 里能看到大量字段" |
| "报错了" | "`kingdee_create_extension_with_python_plugin` 返回 error: `not_initialized: FSUPPLIERNAME 未绑定`" |

**不要从症状直接跳到推测原因**。先把症状钉死。

### 2. 形成 2-3 个假设 · 写出来

**至少 2 个,最多 3 个**。一个假设不够——你需要能对比;超过 3 个说明你对症状理解还不够,回去细化症状。

每个假设要**能被一个工具调用 / 一条查询 / 一个读动作证伪**——如果想不出怎么证伪,那不是好假设,是祈祷。

例(插件注册成功但反查为空):
- **假设 A**:扩展 FID 拼写错了,我 `list_form_plugins` 传的是另一个扩展的 FID
- **假设 B**:插件写进了 `FKERNELXML` 但 parser 读出来时漏掉了某个属性(例如 `PlugInType` 没写)
- **假设 C**:事务 rollback 了,看起来成功实际未落库

### 3. 逐一查证据 · 用只读工具,不写新内容

**一次只验一个假设**,按你认为**最有可能的那个**先查。每个假设对应一个或几个只读工具调用。

- 假设 A → `kingdee_list_extensions('<parentFormId>')` 看返回里有没有我传给 `list_form_plugins` 的那个 extId
- 假设 B → 直接看 `list_form_plugins` 返回的对象结构,某字段是不是 null
- 假设 C → 用 `kingdee_get_object(<extId>)` 看记录是否存在

每查完一条,**明确声明**"这个假设被证实 / 被排除 / 还不确定",不要默默往下一个走。

### 4. 锁定根因 · 没锁定就别动手

证据收集完,判断能不能指向**唯一一个**根因:

- **能** → 进入第 5 步
- **不能**(所有假设都被排除,或两个都支持) → **回到第 1 或 2 步**:症状没描清、或者假设不够。**不要**按最可能的那个瞎猜动手。
- 实在卡住,**如实告诉用户**"我走完了 X / Y / Z 几个假设都被排除,建议检查 [具体方向,比如客户端日志 / K3 服务日志],单凭工具能查到的信息不足以定位"——这比编一个糊弄过去强得多。

### 5. 再动手 · 只改证据指向的那处

根因锁定后:
- 修的内容**只针对那个根因**,不要"顺手把旁边的也改了"
- 修完**还要做反查**(base-system 硬规则四):证据显示问题已经消失才能说"修好了"
- 写进完成消息:**症状 → 根因 → 修复动作 → 验证结果**,四段齐全。客户 / 顾问能据此复盘和信任。

## 简短示例(精简到 4 行,完整走例见 `python-plugin-index/prompts/debugging`)

- **症状**:`kingdee_register_python_plugin` 返回 success,客户端无反应
- **假设 A/B/C**:挂错父扩展 / 事件签名错 / 客户端缓存
- **证据**:先查 A/B(只读工具能查),再问用户测 C
- **锁定**:B 命中(签名 `BeforeSave_1` 错别字)→ 改 + 重注册 + 反查 + 让用户复测

## 和 base-system 硬规则的关系

- 硬规则二"工具返回即事实" → 查证据时只信工具返回,不信记忆
- 硬规则四"写完必须验证" → 修完之后这一步不能省
- 硬规则六 → **就是**强制你遇异常先加载本 skill
