# 审批流 + Python 条件表达式

K/3 Cloud 的审批是**通用平台级能力**——所有单据都能挂审批流,不限于采购 / 销售。

---

## 基本概念

- **审批流** = 一张 DAG(有向无环图):节点 = 审批人 / 角色,边 = 流转条件
- **实例**:一张单据提交审批后,会在某节点停留等审批,审完流到下一节点
- 审批流程中单据状态:`已创建 → 提交 → 审批中 → 已审核`(或 `已退回`)
- **启用方式**:每种单据(销售订单、采购申请、基础资料…)都有"是否启用审批流"开关

---

## 配置路径

1. 系统服务 → **审批流设计器**(切换到"审批模式")
2. 选目标单据 → 新建审批流版本
3. 拖节点 + 连线:
   - **节点类型**:审批(人 / 角色 / 指定字段指定人)、传阅、通知、结束
   - **流转条件**:从 A 到 B 的条件(一般是 True 直通或 Python 表达式判断)
4. 配**审批处理人**:直接指定角色,或按规则动态(`单据.FSellerId.主管`)
5. 发布 → 生效

---

## Python 条件表达式

审批流里**最强大的点**——流转条件可用 Python 写:

```python
# 单据属性:this 指代当前单据(DynamicObject)
# 常用:
this.FAllAmount > 100000         # 金额 > 10 万
this.FCustId.FNumber == "V001"   # 特定客户编码
this.FDate.Month == 12           # 12 月单据

# 组合
this.FAllAmount > 50000 and this.FCustId.FGroup.FNumber == "VIP"
```

这是**IronPython 运行环境**,和表单插件同源(语法 2.7),但**没有** `self.Model` / `self.View` / `self.Context`——审批引擎注入 `this`(只读当前单据对象)。

---

## 典型需求 → 审批流方案

### "5 万以上销售订单要主管审,10 万以上要副总审"

- 节点 1:销售员(提交后直通)
- 条件 1→2:`this.FAllAmount >= 50000`
- 节点 2:部门主管
- 条件 2→3:`this.FAllAmount >= 100000`
- 节点 3:副总
- 条件 3→END:默认

### "跨多个字段组合判断"

```python
# VIP 客户 + 金额超限 + 非测试单据
(this.FCustId.FGroup.FNumber == "VIP"
 and this.FAllAmount > 200000
 and not this.FBillNo.StartsWith("TEST"))
```

### "需要查历史数据的审批条件"

- 本 Python 表达式**不能查别的单据 / 别的表**(审批引擎限制)
- 需要查历史的 → 只能用**扩展字段 + Python 表单插件在 BeforeSave 时填值**,然后审批条件读这个扩展字段

---

## 处理人动态配置

审批节点的"审批人"可以:
- **固定角色**:如"财务部长"
- **按字段动态**:`单据.FSellerId.上级领导`(组织架构递归)
- **按表达式计算**:Python 表达式返回用户 ID

---

## 常见坑

1. **条件表达式里引用基础资料字段要小心 null**——`this.FCustId.FGroup.FNumber` 如果客户没分组会报 None 错。先判空
2. **审批流版本**——修改流程会生成新版本,**已提交的单据仍走旧版本**。要让老单据也走新流程,得强制重启
3. **单据不启用审批流就流程** = 提交即审核完成(秒过)
4. **高并发审批性能**——大量用户同时审批同类单据时,BOS 队列会拉长,建议审批流设计不要嵌套过多节点

## OpenDeploy v0.1 不工具化

审批流**特别不适合程序化**——它是可视化 DAG,手工拖拽才直观。agent 给用户:
1. 点击路径("系统服务 → 审批流设计器 → ...")
2. 节点 / 条件 / 处理人的建议
3. Python 条件表达式**可以**帮用户写,用户 copy 到 Designer 里就行
