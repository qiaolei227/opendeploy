# 权限方案(BOS) · 主流程

> 🟡 **基于金蝶官方手册侦察的主流程版本**。覆盖 K/3 Cloud 多层权限模型(功能 / 数据 / 字段 / 操作 / 组织 / 角色)以及子系统管理员模式。**OpenDeploy v0.1 不工具化任何权限写入**——权限重度依赖客户组织架构,且写错容易把整个账套封死,产品定位是给方向 + 给配置入口,具体步骤交客户管理员手工完成。

---

## 1. 权限模型分层

K/3 Cloud 是 **角色驱动 + 多维叠加** 的权限模型。同一用户最终能做什么 = 「分配的全部角色权限」按以下 5 个维度做交集 / 并集判定:

| 层 | 控制粒度 | 配置入口 | 说明 |
|---|---|---|---|
| **功能权限** | 业务对象 + 12 种操作(查看 / 新增 / 修改 / 删除 / 审核 / 反审核 / 提交 / 复制 / 引入 / 引出 / 打印 / 附件) | 系统管理 → 授权 → 业务对象功能授权 | 决定"能不能进入这个单据 / 能不能点这个按钮" |
| **数据权限** | 在已有功能权限基础上,进一步限定**哪些数据**(行级,按组织 / 部门 / 业务员 / 自定义表达式) | 系统管理 → 授权 → 业务对象功能授权 → 数据范围 | 必须先有功能权限才能设数据权限,二者不是平行 |
| **字段权限** | 字段级查看 / 编辑 | 系统管理 → 授权 → 字段授权 | 默认全部"有权",仅设"无权 / 禁用" |
| **操作权限** | 单条操作的可见性 + 校验规则 + 操作日志 | BOS 设计器 → 操作列表 → 其他控制 | 操作权限挂在「操作」上,功能权限挂在「业务对象」上,二者交叉 |
| **组织权限** | 用户能在哪些组织(账套 / 业务组织)下工作 | 用户 → 用户组织 / 角色 → 授权范围 | 授权范围决定该角色在哪些组织下生效;不在范围内的组织一律隐藏 |

**来源**: <https://help.open.kingdee.com/dokuwiki/doku.php?id=权限控制与应用> · fetched 2026-04-23

## 2. 用户与角色绑定

### 用户(User)
- **控制粒度**: 系统操作者实体,登录账号
- **配置入口**: 系统管理 → 用户
- **典型场景**: 新员工入职 → 建用户 → 关联到 1 个或 N 个角色
- **常见坑**:
  - 用户未"启用"则登不上;启用 / 禁用是状态字段,不是删除
  - 一个用户可挂多角色,**权限取并集**——给临时权限时建议另建角色而不是直接改原角色
  - 用户必须关联"业务员"基础资料才能在数据规则里被识别为"当前用户对应的业务员"
- **OpenDeploy 工具覆盖**: ❌
- **来源**: <https://help.open.kingdee.com/dokuwiki_std/doku.php?id=用户> · fetched 2026-04-23

### 角色(Role)
- **控制粒度**: 权限的承载实体——所有功能 / 数据 / 字段 / 操作授权都挂在角色上,不直接挂用户
- **配置入口**: 系统管理 → 角色
- **类型**:
  - **系统管理员角色**(可设授权范围,典型用法是"子系统管理员")
  - **普通角色**(只能被授权,不能授权别人)
- **典型场景**: 按岗位建角色(采购员 / 采购主管 / 财务 / 销售内勤),用户挂一个或多个
- **常见坑**:
  - 角色一旦被多个用户引用,改权限**会立刻影响所有挂载用户**——生产环境改前先用"权限报表"评估影响面
  - 删除角色不会删用户,但用户会失去该角色的全部权限
- **OpenDeploy 工具覆盖**: ❌
- **来源**: <https://help.open.kingdee.com/dokuwiki_std/doku.php?id=角色> · fetched 2026-04-23

## 3. 角色权限继承

🟡 **K/3 Cloud 不是经典的"角色继承树"模型**——没有"父角色 / 子角色"自动继承。继承靠两条机制实现:

1. **批量功能授权**: 把一个角色的全部权限项一键复制到另一个角色(系统管理 → 授权 → 批量功能授权)
2. **子系统管理员授权模式**: 把"系统管理员"权限拆细——由总管理员先指派"采购子管理员",采购子管理员只能在「采购」子系统范围内授权,不能越权改其他子系统。这是 K/3 Cloud 实现"分级管理"的官方方案,不是真正的角色继承。

- **典型场景**: 集团公司 IT 一个总管理员 + 各分公司业务子管理员各自管自己的子系统
- **常见坑**:
  - 子系统管理员授权一旦给出去,**收回时不会自动回收他已经授权出去的下游角色权限**——需要手动审计
  - 没有"角色 A 继承自角色 B"这种自动同步机制,B 改了 A 不会跟着变
- **OpenDeploy 工具覆盖**: ❌
- **来源**: <https://help.open.kingdee.com/dokuwiki_std/doku.php?id=子系统管理员功能授权模式> · fetched 2026-04-23

## 4. 数据权限规则(动态表达式)

数据权限是 K/3 Cloud 权限模型里**最常出问题**的一块。

- **控制粒度**: 行级。在已有功能权限基础上,通过"数据规则"过滤出该角色能看到的行
- **配置入口**: 系统管理 → 授权 → 业务对象功能授权 → 选定权限项 → 数据范围
- **支持的权限项**: 查看 / 修改 / 删除 / 审核(其他权限项不参与数据规则过滤)

### 规则写法
两种形态:
1. **常量条件**: `部门名称 = "采购一部"`(写死)
2. **变量条件**(推荐,大幅减少规则数量):
   - `当前组织` —— 字段类型为组织
   - `当前用户` —— 字段类型为用户
   - `当前用户对应的部门`
   - `当前用户对应的业务组`
   - `当前用户对应的业务员`

举例:"我的销售订单" = 业务单据(销售订单).创建人 = 当前用户。一条规则覆盖所有用户,不必每人建一份。

### 关键限制
- ❌ **单据体字段不支持**(物料 / 仓库 / 行金额等)——默认情况下数据规则只对单据头字段有效。需启用「单据列表启用单据明细数据权限控制」参数后才能用单据体
- ❌ **不支持联动控制**——一个数据规则只对一个业务对象生效,不会自动传播到下推 / 关联单据。例:对销售订单设了"只看本部门",发货通知**不会自动跟着只看本部门**,得对发货通知再设一遍
- 🟡 **基础资料联动**(V7.7+): 启用"基础资料数据范围联动应用到业务对象"参数后,业务单据会自动只显示有权限的物料 / 客户 / 供应商
- 🟡 **打开含无权限明细的单据**: V7.7 起可设参数禁止
- 🟡 **基础资料直接授权**(V8.2+): 不走数据规则,直接选物料 / 客户列表勾给某员工

- **典型场景**: 多事业部隔离销售数据 / 跨地区分公司只看本地仓库
- **常见坑**:
  - 漏设单据体规则导致用户"看得到表头但看不到行"——表现是单据空白
  - 多角色叠加时数据规则取**并集**,看到的比预期多
  - 建数据规则前先把对应权限项设为"有权",否则规则不生效
- **OpenDeploy 工具覆盖**: ❌

## 5. 字段权限(含扩展字段处理)

- **控制粒度**: 字段级。两个开关:**查看权** + **编辑权**
- **配置入口**:
  1. **运行时授权**: 系统管理 → 授权 → 字段授权 → 选角色 → 勾选「显示全部业务对象」 → 业务对象列表选字段 → 设"无权 / 禁用"
  2. **设计期开关**: BOS 设计器 → 编辑 → 权限控制设置 → 勾「控制字段权限」(不勾则该单据所有字段不进字段授权列表)
- **典型场景**: 财务能看销售订单但看不到「采购成本」字段;销售内勤能看「客户联系方式」但不能改
- **扩展字段处理**:
  - BOS 扩展或 BOS 自定义新加的字段**默认有权**——给所有角色暴露
  - 新加敏感字段后,必须**主动**到字段授权里设"无权",否则全员可见
  - 扩展对象上的新字段在「字段授权」列表里独立出现,不继承基础对象的字段权限设置
- **常见坑**:
  - BOS 设计器中没勾「控制字段权限」开关,字段授权页根本看不到这个单据的字段
  - 字段权限不影响 SQL 直查,只影响 UI / 报表展示——客户做了字段权限不等于数据安全
  - "用户业务对象字段权限报表" 可批量审计某用户在某业务对象下所有字段的实际权限,出问题先查这张报表
- **OpenDeploy 工具覆盖**: ❌
- **来源**:

## 6. 操作权限

- **控制粒度**: 单条「操作」(新增 / 修改 / 删除 / 审核 / 反审核 / 提交 / 复制 / 引入 / 引出 / 打印 / 附件 / 自定义按钮)
- **配置入口**:
  1. **设计期声明**: BOS 设计器 → 操作列表 → 选操作 → 「其他控制」页签 → 设权限项 / 校验规则 / 是否写操作日志
  2. **运行时授权**: 系统管理 → 授权 → 业务对象功能授权 → 勾权限项为"有权"
- **预置 12 种**: 查看 / 复制 / 删除 / 新增 / 修改 / 提交 / 审核 / 反审核 / 附件 / 引入 / 引出 / 打印
- **自定义权限项**: BOS 设计器 → 权限项管理 → 新增,然后在操作的"其他控制"里挂载
- **典型场景**: 销售员能新增 / 修改销售订单,但只有销售主管能审核
- **常见坑**:
  - 自定义按钮没在权限项里挂载 → 该按钮**对所有人可见可点击**,等于裸奔
  - 反审核默认权限和审核**分开**,常见漏给反审核权限导致客户改不动单据
  - 操作权限和数据权限是两层——有审核权 + 数据规则限定"只能审本部门" = 对部门外的单据看得到但点不动
- **OpenDeploy 工具覆盖**: ❌

## 7. 组织权限

- **控制粒度**: 用户能登录到哪些**组织**(账套 / 业务组织 / 核算组织)下,以及在那个组织下生效哪些角色
- **配置入口**:
  1. 用户 → 用户组织: 选这个用户允许进入的组织清单
  2. 角色 → 授权范围: 这个角色在哪些组织生效
- **多组织模型**:
  - 必须先购买"多组织"软件授权才能启用,在「多组织实施向导」里逐步建组织树
  - 业务组织 ≠ 核算组织 ≠ 销售组织——可独立设,也可合并;典型实施踩坑点
- **典型场景**:
  - 集团 + 子公司:用户 A 是集团采购,挂"采购员"角色,授权范围 = 全部子公司;用户 B 是华南分公司采购,授权范围 = 华南
  - 同一角色在不同组织下数据隔离,自动靠"当前组织"变量实现
- **常见坑**:
  - 用户没加到目标组织,登录后该组织在切换器里**根本不显示**——常被误判为"权限丢了"
  - 角色授权范围设"全部组织"是危险默认值,新角色应明确指定
  - 组织变更(如分公司从华南挪到华东)需走「组织变更」流程,直接改 SQL 会让历史单据报表错乱
- **OpenDeploy 工具覆盖**: ❌

## 8. 权限委托 / 临时授权

🔴 **官方手册侦察未找到「权限委托」或「临时授权」的独立模块**(2026-04-23 搜索 `help.open.kingdee.com` 域名,无匹配文档)。

K/3 Cloud 实践中"临时把权限给某人"的常见替代方案:

1. **临时挂角色**: 把目标用户加到「临时审批」角色 → 完事再取消挂载。最直接,无审计链路
2. **子系统管理员模式**: 让总管理员临时指派"子系统管理员",对方能在指定子系统范围内自行处理授权。适合"出差期间让副手代管"
3. **代理审批 / 工作流转交**: 不在权限模块,在工作流里设置代理人——和"权限委托"是两回事,只代具体审批任务

**方向性建议**(非实证):
- 客户问"出差代审"——先指引用工作流的「代理人」/「转交」功能,不是改权限
- 客户问"周末让张三临时管系统"——指引子系统管理员模式
- 客户问"做权限委托审计"——告知 K/3 Cloud 没原生委托模块,需要客户自己建审计流程或上 BOS 自定义

- **OpenDeploy 工具覆盖**: ❌

## OpenDeploy v0.1 工具覆盖总览

❌ **整个权限模块全部不工具化**。原因:
1. 权限设计高度依赖客户组织架构 + 岗位划分,无通用模板
2. 权限写错容易把账套封死(典型:误改系统管理员角色),恢复成本高
3. 多角色多组织叠加的实际效果难以静态推算,需 UAT 验证
4. 数据规则是带 SQL 表达式的运行时过滤,agent 写出的规则不在 OpenDeploy 写白名单内

**OpenDeploy 现阶段定位**: 给方向 + 指出配置入口 + 提示踩坑,**不写任何权限相关的元数据**。
