# K/3 Cloud 插件类型全谱

K/3 Cloud BOS 有 **6 种插件类型**,OpenDeploy v0.1 **只自动化表单插件(FormPlugins)**。其他 5 种用户要做的话只能手工 BOS Designer 注册。

---

## 类型总表

| 插件类型 | 元数据节点 | 触发位置 | 典型场景 | OpenDeploy v0.1 |
|---|---|---|---|---|
| **表单插件** | `<FormPlugins>` | 客户端 UI 层 | 保存前校验、字段联动、按钮行为 | ✅ 自动化 |
| **列表插件** | `<ListPlugins>` | 列表页面 | 批量操作、列表过滤扩展、导出定制 | ❌ 手工 |
| **操作插件** | `<OperationServicePlugins>` | 服务端操作 | 审核 / 反审核拦截、下推校验、删除前判断 | ❌ 手工 |
| **转换插件** | `<ConvertPlugins>` | 单据转换时 | 下推字段映射复杂逻辑、转换前校验 | ❌ 手工 |
| **打印插件** | `<PrintPlugins>` | 打印时 | 打印数据定制、条形码 / 二维码生成 | ❌ 手工 |
| **报表插件** | `<ReportPlugins>` | 报表查询时 | 报表数据源扩展、过滤条件补充 | ❌ 手工 |

## 列表插件

**触发位置**:业务单据的**列表页面**(打开一个业务单据类型时看到的那个"订单列表"页)

**典型场景**:
- 批量给选中单据打标记
- 增加列表右键菜单(比如"给选中客户发短信")
- 列表数据双击跳转定制

**技术形态**:IronPython 或 DLL,基类 `AbstractListPlugIn`,常用事件 `Loaded` / `ItemDoubleClick` / `BarItemClick`

**为什么 v0.1 不工具化**:需求比表单插件低一个数量级,优先级后置

---

## 操作插件(重要但不覆盖)

**触发位置**:**服务端**操作(不是 UI 层)。比如"审核"操作——客户端点"审核"按钮,请求走到服务端的 `AuditService`,操作插件在这里拦截。

**典型场景**:
- **审核拦截**:金额超某值不能审核(必须副总手动审)
- **反审核拦截**:已下推的单据不能反审
- **下推前校验**:下推时检查库存充足
- **删除拦截**:已关联其他单据的不能删

**为什么重要**:很多"校验"需求,实施顾问以为是表单插件的事,**其实要写操作插件**——因为真正的动作发生在服务端,客户端拦截不住(用户可以绕过 UI 直接调 API)。

**识别话术**:当用户说"审核时拦截" / "下推时校验" / "删除前提醒",告知他们这需要操作插件,OpenDeploy v0.1 不支持,建议:
1. 手工 BOS Designer 写操作插件
2. 或者把时机前移到"保存前"(表单插件能做),但 UI 层拦截比服务端弱

**技术形态**:DLL(C#)为主,IronPython 也可,基类 `AbstractOperationServicePlugIn`

---

## 转换插件

**触发位置**:单据下推时(`销售订单 → 发货通知` 这种)

**典型场景**:
- 下推前校验源单据状态
- 下推时做复杂字段映射(标准字段映射做不到的)
- 下推后调用外部系统同步

**技术形态**:`AbstractConvertPlugIn`

**为什么 v0.1 不覆盖**:转换规则的字段映射 + 业务规则已经能覆盖 90% 场景,插件是余下 10% 的兜底

---

## 打印插件

**触发位置**:用户点"打印"后,打印数据生成阶段

**典型场景**:
- 打印时生成二维码 / 条形码图片
- 打印数据里加"合计大写金额"
- 根据单据状态选不同打印模板

**技术形态**:`AbstractPrintPlugIn`

---

## 报表插件

**触发位置**:内置报表查询时

**典型场景**:
- 报表数据源加自定义过滤条件
- 报表数据计算列
- 报表展示前预处理

**技术形态**:`AbstractReportPlugIn`

## agent 话术模板

当识别到**非表单插件**需求:

```
用户:"审核时校验订单金额"

Agent:
"'审核拦截' 属于 K/3 Cloud 的操作插件(OperationServicePlugin),和表单插件(FormPlugin)不是一类。

- 表单插件触发在客户端 UI 层,适合'保存前'、'字段变'等场景
- 操作插件触发在服务端,'审核 / 反审核 / 下推 / 删除' 这类真操作走这里

OpenDeploy v0.1 只自动化表单插件。对你这个需求有两条路:

1. **改成保存前校验**:把校验时机提到保存,保存通过就说明金额合规。缺点:用户有可能在 UI 层绕过(改值后不保存直接审核)
2. **手工写操作插件**:在 BOS Designer 里注册 `OperationServicePlugin`,挂在'审核'操作上。需要 C# / Python 基础,我可以帮你写代码骨架但注册要你自己来

你想走哪条?"
```

**诚实给出边界**,不要硬用表单插件模拟审核拦截——会让客户环境出现"UI 层有拦截、绕过就没了"的安全漏洞。
