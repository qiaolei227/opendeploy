# BOS 定制层级选择指引

当一个需求"标准功能没有,但 BOS 能做"时,问自己下面这个决策树,按顺序选**最轻的能解的那层**。

---

## 决策树

```
需求描述: ______

Q1: 是否只需要"在单据上多记录一个字段的值"?(纯存储,不涉及计算 / 联动 / 校验)
├─ 是 → 【扩展字段】(references/custom-fields)
│       OpenDeploy v0.1 ❌ 手工 BOS Designer
└─ 否 → Q2

Q2: 是否只需要"字段 A 变了,字段 B 自动算出来"?(纯公式,不带复杂逻辑)
├─ 是 → 【业务规则 / 公式】(references/business-rules)
│       OpenDeploy v0.1 ❌ 手工
└─ 否 → Q3

Q3: 是否只需要"从 A 单据按规则生成 B 单据"?(字段对应映射)
├─ 是 → 【转换规则】(references/convert-rules)
│       OpenDeploy v0.1 ❌ 手工
└─ 否 → Q4

Q4: 是否只需要"XX 条件的单据要多一级审批"?
├─ 是 → 【审批流 + Python 条件】(references/approval-workflow)
│       OpenDeploy v0.1 ❌ 手工
└─ 否 → Q5

Q5: 需要事件驱动的业务逻辑?(BeforeSave 校验 / 字段联动 / 按钮行为)
├─ 是 → 【Python 表单插件】
│       OpenDeploy v0.1 ✅ kingdee_create_extension_with_python_plugin
│       详见 k3cloud/python-plugin-index
└─ 否 → 回 solution-decision-framework 的第 4 优先级(DLL / 告诉用户超出范围)
```

---

## 选层原则

### 优先轻的 → 重的(可维护性)

- 扩展字段 < 业务规则 < 转换规则 < 审批流 < 插件
- 越轻的方案,日后交接 / 升级 / 排障越省事
- 一堆 Python 插件 = 技术债,顾问辞职后客户没人能维护

### 不要"混用"同一个问题

比如"字段 B = 字段 A * 2":
- ✅ 业务规则一条搞定
- ❌ 写 `DataChanged` 插件

业务规则能做的就别开 Python——维护成本差 10 倍。

### 插件只在"事件驱动"时才该用

插件的核心价值是**响应事件**(保存前、点击后、字段变后)。如果需求描述里没有"什么时候触发"这个词,几乎可以确定**不该用插件**。

---

## 常见误判(顾问或 agent 容易选错层)

### ❌ "加个字段" → 写 Python 插件

正解:扩展字段。Python 不是用来存数据的,单纯加字段走扩展字段。

### ❌ "金额=数量*单价自动算" → 写 `DataChanged` 插件

正解:业务规则 / 公式。BOS 内置公式引擎,性能好可视化编辑。

### ❌ "销售订单下推发货通知时带上客户地址" → 写插件

正解:转换规则加字段映射。本来就是下推的事。

### ❌ "1 万以上审批,5 万以上加副总审" → 写插件

正解:审批流 + Python 条件表达式。BOS 审批设计器天然支持。

### ❌ "校验信用额度不能超" → 扩展字段 / 业务规则

正解:**这个才是 Python 插件的场景**——BeforeSave 事件阻断,业务规则只能算值不能取消保存。

---

## 混合场景

复杂需求可能需要多层组合。比如:
- 扩展字段 `F_VIP_TAG` 标记客户是否 VIP(字段层)
- 业务规则:VIP 客户订单自动 95 折(规则层)
- 审批流:折扣率低于 9 折要审批(审批层)
- Python 插件:保存前校验 VIP 客户当月下单次数不超过 5 单(插件层)

这种时候**不要把 Python 插件当主角**——它是最后一道把关,主体逻辑在更轻的层。层次越清晰,维护越稳。

## 跟用户沟通模板

当你识别出合适的 BOS 层级但 OpenDeploy 不工具化时,告知用户:

> "这个需求在 BOS 层面属于 **<层级名>**。BOS Designer 里的操作步骤是:
> 1. <步骤>
> 2. <步骤>
> 3. <步骤>
>
> 当前 OpenDeploy 工具暂不自动化这一步,建议你按上面手工配置一次。配完 F5 刷新即可生效。"

**给操作步骤,不要只说"你去 Designer 做"**。越具体越体现顾问价值。
