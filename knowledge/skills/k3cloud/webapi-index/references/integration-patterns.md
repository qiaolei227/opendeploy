# K/3 Cloud 集成模式选型

<!-- 来源:vip.kingdee.com 集成汇总 + 社区实施文章;实证状态:🟡 主流程,具体阈值 / 方案选型非客户环境实测 -->

客户说"要和 XXX 对接"时,**别上来就给端点列表**——先分清**哪种集成模式**。模式错了,端点再对也跑不起来。

---

## 4 种集成模式

| 模式 | 特征 | 典型场景 | 复杂度 |
|---|---|---|---|
| **实时拉取**(Pull) | 外部系统主动查 K/3 Cloud | BI 报表 / 数据看板 / 移动 APP 查订单 | 低 |
| **实时推送**(Push) | 外部系统主动写 K/3 Cloud | 电商下单 / POS 结算 / OA 审批回写 | 中 |
| **批量同步**(Batch) | 定时双向同步,大批量 | 历史数据迁移 / 日结 / 每月报表 | 中 |
| **事件驱动**(Event) | K/3 Cloud 内部事件触发外部动作 | 审核后发钉钉消息 / 出库推 WMS / 变更推 BI | 高 |

---

## 1. 实时拉取(Pull)

### 特征

- 外部系统**每次需要时**发 WebAPI 查一次
- K/3 Cloud 是**被动**的数据源
- 数据量一般小(查单条 / 查分页列表)
- 时延要求高(秒级)

### 端点

主要用 `ExecuteBillQuery`(列表)+ `View`(详情)+ `GetSysReportData`(报表)。详见 `common-endpoints`。

### 适用场景

- 客户的 CRM / BI 工具需要实时看销售订单状态
- 移动 APP 查库存 / 查价格
- 第三方客服系统按客户号查订单

### 陷阱

- **大表分页必须带 `OrderString`**——没有排序的 OFFSET 分页结果可能重复 / 漏
- **查询字段权限**——管理员用户可能查不到价格 / 成本(无报表角色),换业务用户登录
- **基础资料字段要用 `.FNumber`**——直接拿 FCustId 只是 GUID 内码,拿 FCustId.FNumber 才是人读得懂的客户编号

### 话术

> "这是典型的 Pull 模式——你们 BI / CRM 需要时查 K/3 Cloud。用 `ExecuteBillQuery` 端点拉列表,`View` 拉详情。**不要硬要求秒级**,列表页建议缓存 5 分钟;K/3 Cloud 不是高 QPS 的接口服务。"

## 2. 实时推送(Push)

### 特征

- 外部系统**主动写**到 K/3 Cloud
- 事件触发(用户下单 / 审批通过 / 签到)
- 要求立即落账
- **易失败**(业务规则多 / 字段联动 / 主数据校验)

### 端点

`Save` + `Submit` + `Audit`(拆开调用,别用 `IsAutoSubmitAndAudit:true`)。

### 适用场景

- **电商订单进 K/3 Cloud**:淘宝 / 天猫 / 京东订单流水 → 销售订单
- **POS 小票进财务**:门店现金销售 → 销售出库单 + 收款单
- **OA 审批回写**:OA 完成审批 → K/3 Cloud 审核对应单

### 陷阱

- **主数据不一致**——外部系统的"客户"编码和 K/3 Cloud 的 `FCustId.FNumber` 不一致,必须**先建主数据映射表**
- **并发下单撞单号**——让 K/3 Cloud 自动生成编号(`FBillNo` 留空),不要外部生成
- **网络抖动导致重复推**——**必须**在外部业务 ID 和 K/3 Cloud 内码之间建幂等表(`外部订单号 → FID`),重推前查一次
- **定价服务冲突**——外部推的单价被 K/3 Cloud 自动定价覆盖成 0,见 `error-handling`

### 架构建议

**在外部系统和 K/3 Cloud 之间加一层"集成缓冲表"**:

```
外部系统 → 集成缓冲表(待推 / 已推 / 失败) → WebAPI → K/3 Cloud
```

好处:
- 失败可**重推**(看缓冲表状态)
- 峰值削峰(缓冲表里慢慢消化)
- **审计可追溯**——每条推送留日志

### 话术

> "Push 模式第一原则:**一定要做幂等**。外部订单号绑 K/3 Cloud 内码,推送前查一次映射表。失败**不要自动重试**,留一张错误表给人工修。主数据(客户 / 物料)**先对齐**再推,不然 95% 的失败都是主数据问题。"

## 3. 批量同步(Batch)

### 特征

- 定时触发(每小时 / 每日 / 每月)
- 大批量(几百到几万条)
- 时延宽松(小时级)

### 端点

`BatchSave`(推入)+ `ExecuteBillQuery`(拉出),配合客户端分批逻辑。

### 适用场景

- **历史数据迁移**——老系统的订单 / 单据搬进 K/3 Cloud
- **日结同步**——白天收集的数据夜间批量入账
- **跨系统对账**——WMS / 财务系统互对库存 / 应收

### 关键参数

参考 `error-handling` 里的限流表:

- 单次 `BatchSave` ≤ 20 条
- `BatchCount` ≤ 10(推荐 ≤ 5)
- 每个执行计划周期处理 200 条,跑 5 分钟内结束
- 客户端线程 2-10 个

### 架构建议

**三段式**:

```
Stage 1: 抽取(Extract)——从外部系统抽到中间表,字段简化
Stage 2: 转换(Transform)——在中间表补主数据映射 / 格式转换
Stage 3: 加载(Load)——分批调 BatchSave,失败留痕
```

成熟工具:**DataX / Kettle / qeasy / 轻易云 / 集简云**——都是做 K/3 Cloud 批量集成的 ETL 工具,客户如果批量复杂建议直接上这些,不要从零写。

来源:[金蝶云星空销售出库单高效集成](https://www.cnblogs.com/pychcyg/p/18530423) + [使用轻易云到金蝶数据传输](https://www.qliang.cloud/dataintegration/a94ab159-a598-337e-84bc-7c6f1628d912) fetched 2026-04-23

### 陷阱

- **别搞"全量覆盖"同步**——K/3 Cloud 的业务数据有下游关联(库存已出 / 应收已开),全量覆盖会连带下游污染
- **增量标识字段**——用 `FModifyDate` 或**外部系统的 updated_at** 做增量,不要用 `FCreateDate`(改单不更新它)
- **时区**——K/3 Cloud 默认服务器时区,外部系统如果 UTC 要转

### 话术

> "批量 ≥ 几千条就别自己从零写集成,找 **轻易云 / 集简云 / qeasy / 金蝶官方 ETL** 这类工具,他们已经处理了幂等 / 限流 / 主数据映射这些通用问题。我们主要帮你设计数据流和字段映射,集成工具的参数配置来落地。"

## 4. 事件驱动(Event)

### 特征

- K/3 Cloud 里发生事件(保存 / 审核 / 状态变)**触发外部动作**
- 不是外部主动查,也不是批量定时
- 要求近实时(秒到分钟)

### 端点

**WebAPI 不直接支持事件驱动**——WebAPI 只能被调用,不能主动通知外部。实现方式有 3 种:

### 方式 A:服务端插件回调

在 K/3 Cloud 服务端写 **Python / DLL 操作插件**,在 `AfterAudit` 等事件里**主动 HTTP 调用外部系统**。

```python
# 伪代码 — 操作插件 AfterAudit
def AfterAudit(self, args):
    billNo = self.Model.GetValue('FBillNo')
    requests.post('https://外部系统/webhook', json={'bill': billNo})
```

**优点**:事件触发精确,延迟最小。
**缺点**:插件同步调用外部会**拖慢 K/3 Cloud 自身**——外部超时 = 审核卡死。
**对策**:写日志到一张"待发送"表,用**单独线程池**或**消息队列**异步发。

参考 `k3cloud/python-plugin-index`,但注意**操作插件**不在 OpenDeploy v0.1 工具覆盖范围,需用户在 BOS Designer 手工注册。

### 方式 B:轮询补偿

外部系统**高频轮询** K/3 Cloud 的 `ExecuteBillQuery`,按 `FModifyDate > last_check` 拉变更。

**优点**:K/3 Cloud 不用改,零侵入。
**缺点**:延迟 = 轮询周期;频繁调用给 K/3 Cloud 压力;K/3 Cloud 侧无法区分"真的变了"和"只是时间戳更新"。

### 方式 C:消息队列中转

K/3 Cloud 插件只写消息到 **RabbitMQ / Kafka / RocketMQ**,外部系统订阅队列消费。

```
K/3 Cloud 插件 → MQ → 外部系统消费者
```

**优点**:K/3 Cloud 只负责投消息(快),外部系统慢也不拖垮,天然削峰,消费失败可重试。
**缺点**:架构复杂,需要独立的 MQ 基础设施。**适合体量大 / 多个下游订阅方**的场景。

### 话术

> "事件驱动不是 WebAPI 本身的能力,要在 K/3 Cloud 服务端写**操作插件**,在审核后主动推外部。**但插件里别直接 HTTP 调用**——要写到队列或本地表异步发,否则外部慢会拖死审核。如果对接方多建议上 MQ(RabbitMQ / Kafka),单接一个可以插件直接推。"

## 模式选型决策表

收到需求时问这几个问题:

| 问题 | 答案倾向 | 推荐模式 |
|---|---|---|
| 谁主动发起? | 外部查 K/3 Cloud | Pull |
| | 外部写 K/3 Cloud | Push |
| 有没有定时节奏? | 实时(秒级) | Pull / Push |
| | 每日 / 每月 | Batch |
| | K/3 Cloud 事件触发 | Event |
| 数据量? | < 100 条 / 次 | Pull / Push |
| | > 1000 条 / 次 | Batch |
| 能容忍延迟吗? | 秒级要求 | Pull / Push / Event(插件直推) |
| | 分钟级 | Event(MQ) |
| | 小时级 | Batch |
| 下游对接方数量? | 1-2 个 | 插件直推即可 |
| | 多个 / 未来会增加 | MQ |

## 红线:什么时候绝对不能走 SQL 直连

**客户或其他乙方**可能提出"直接连数据库读 / 写快"——**必须反对**,原因:

1. **业务表有复杂联动**:销售出库单不是 `T_SAL_OUTSTOCK` 一张表,背后有库存扣减 / 应收生成 / 成本结转等十几个关联动作,SQL 绕过 = **数据污染**
2. **BOS 审计 / 插件 / 权限全失效**——SQL 直写没经过 BOS 引擎,审批流 / 字段校验 / 权限检查**全被跳过**
3. **金蝶服务条款**——生产环境 SQL 直写业务表**违反金蝶许可协议**,出问题客户找金蝶求助会被打回
4. **OpenDeploy 架构硬红线**——产品只允许读元数据表 + 写 BOS 扩展元数据,业务表 SQL 读都不允许,更别说写

**如果客户坚持要快,告诉他:**

- 查:`ExecuteBillQuery` 调优 `FieldKeys`(少选列)+ `FilterString`(加索引过滤)就够快,一般 1000 条 / 秒
- 写:用 `BatchSave` + `BatchCount=5` + 客户端 5 线程,可到 100-500 条 / 秒
- 真的还不够快,**是业务架构问题,不是 API 问题**——要么削峰(加 MQ),要么分账套(不同业务到不同数据中心)

## 什么时候让客户用 ESB / iPaaS

**客户自建集成层代价太高的场景**,推荐用**集成平台(iPaaS)**:

- **下游系统 > 3 个**:一对多变 mesh,自建维护代价指数级上升
- **每天 > 10 万条事务**:自建需要考虑队列 / 限流 / 补偿 / 告警,iPaaS 都有
- **多个跨厂商系统互联**:SAP / 金蝶 / 用友 / 钉钉 / 企微 混合,iPaaS 有现成连接器

**主流 iPaaS(金蝶生态)**:
- **轻易云**(qliang.cloud)——金蝶深度集成
- **集简云**(jijyun.cn)——无代码连接
- **qeasy**——做 K/3 Cloud 的多
- **钉钉宜搭 / 腾讯 iPaaS** — 私有云 / 钉钉生态

OpenDeploy 作为实施交付智能体,**不做 iPaaS 连接器**——我们的定位是单客户的设计 + 部署,多客户的集成规模化是 iPaaS 的领域。**遇到这类需求告诉用户"上 iPaaS 工具"比"自建"更合适**。

## OpenDeploy 工具覆盖

| 模式 | OpenDeploy 可帮做 | 用户 / 客户团队做 |
|---|---|---|
| Pull | 帮用户**设计查询字段** / 写 `FilterString` / 调优索引 | 实际的查询调用 |
| Push | 帮用户**设计字段映射** / 生成 `Save` JSON 模板 | 实际的推送调用 / 重试 / 幂等 |
| Batch | 帮用户**评估数据量 / 选 ETL 工具** | ETL 配置 / 运行 |
| Event | 帮用户**写服务端操作插件**(如果是 Python 表单插件;操作插件不工具化)| 插件注册 / MQ 搭建 / 下游消费 |

## 临时话术(集成需求进来的第一句)

> "这个需求是**哪种集成模式**?我需要先分清:
> - **谁发起**(外部查 K/3 / 外部写 K/3 / K/3 事件触发外部)?
> - **实时还是批量**(秒级 / 分钟级 / 小时级)?
> - **数据量多大**(每天几条还是几万条)?
>
> 选错模式端点再对也跑不起来。定好模式后我给你对应的端点组合 / 架构建议 / 坑位清单。"
