# K/3 Cloud 数据对不上诊断 🟡

<!-- 来源:help.open.kingdee.com / vip.kingdee.com / m.heshuyun.com / blog.csdn.net 公开内容综合;实证状态:🟡 常见场景索引,具体解决方案以客户环境为准 -->

覆盖"数据对不上"这一大块——**系统跑通了,但数字不符合用户预期**。包括库存账实不符、凭证生成不全、账龄口径不一致、报表为空、现金流量表数据缺失。

**核心纪律**:数据对不上的问题**几乎从不是"系统 bug"**,而是:

1. **业务录入不规范**(单据日期错 / 补单不按顺序 / 漏单)
2. **配置差异**(科目 / 核算维度 / 报表取数方案的口径)
3. **报表查询条件不同**(两张报表底层算法不同)
4. **期间 / 权限过滤**(用户看到的是部分数据)

所以 agent 的姿势:**把数字从多个维度拉出来对齐**(SQL vs 报表 vs 单据明细),而不是"系统算错了"。走 `common/systematic-debugging` 五步——症状钉死前不要下结论。

## 1. 库存账实不一致

### 场景 1.1:仓库实物盘点 vs 系统库存数量不符

- **症状**:盘点表上物料 A 仓库 X 有 100 件,系统"即时库存"显示 95 件(或 110 件)
- **常见原因**:
  1. **入库单 / 出库单日期录错**——入库单日期晚于出库单,导致中间某时刻账面负库存
  2. **漏单 / 补单没按顺序**——先做了出库,入库单后补,业务日期没倒回去
  3. **跨组织调拨未完整**——调出单审核了但调入单未审核
  4. **单位转换错**——基本单位数量和辅助单位数量一正一负(如按箱出,按件入)
  5. **生产领料 / 退料**未如实记录
  6. **盘盈盘亏单**没做(系统默认两边一致,实际有差异未调平)
- **诊断步骤**:
  1. 用 `kingdee_*` 工具查该物料 + 该仓库的**收发明细**,按日期排序
  2. 逐笔对账:每一笔入库 / 出库的日期 / 数量 / 单据号是否和实物流对得上
  3. 特别注意:"即时库存"是实时累加,"结存库存"是按期间计算,**两者算法不同**
- **解决方向**:
  - 定位到具体哪笔单据异常,走**反审核 → 改日期 / 数量 → 重审核**
  - 如果改不动(已关账),用**盘盈 / 盘亏单**在当期调平
  - 批量问题建议客户**先做一次全盘**,建立新基线后严格管控单据日期
- **链到**:如果涉及多组织调拨,看 `operation-failures` §1(下推失败);如果库存关账报负库存,看 `standard-errors` §4.1
- **来源**:[即时库存正常,但是账面负结存原因分析](https://vip.kingdee.com/article/12215) · [第1-15期 金蝶云·星空-供应链云问答精选](https://vip.kingdee.com/article/297868445403148544) fetched 2026-04-23

### 场景 1.2:即时库存 vs 结存库存两表对不上

- **症状**:用户同时看"即时库存"和"库存余额表",物料 A 数字不同
- **常见原因**:**这不是 bug,是算法差异**
  - **即时库存**:实时累加,截止到当前这一秒
  - **库存余额表**:按**期间**计算(上期末 + 本期入 - 本期出),受期间参数影响
- **诊断步骤**:问用户选的**查询日期**——即时库存是当前时刻,余额表默认是当期末
- **解决方向**:告诉用户两张表**设计就是不同口径**,按需求选:看实时状态用即时库存,看期间结存用余额表
- **来源**:[第1-15期 金蝶云·星空-供应链云问答精选](https://vip.kingdee.com/article/297868445403148544) fetched 2026-04-23

## 2. 凭证生成不全 / 失败

### 场景 2.1:业务单据审核了但没生成凭证

- **症状**:销售出库 / 采购入库 / 应收应付已审核,但"凭证生成"按钮点后报"生成 0 条"或提示错误
- **常见原因**:
  1. **凭证模板的分录行生成条件不满足**——模板里设了条件(如"仅走国内贸易"),当前单据不匹配
  2. **科目取不到**——科目影响因素(核算维度)没全设,或基础资料未分配
  3. **借贷不平**——借方和贷方的分录行生成条件不对称,导致一边取到值一边取不到
  4. **金额为零**的分录被跳过
  5. **定时任务没跑**——如果配了自动生成凭证,服务端计划任务没启动
- **诊断步骤**:
  1. 让用户打开**凭证模板**看条件配置
  2. 查"**凭证生成报告**"——金蝶会列出每张单据的生成状态 + 失败原因
  3. 用 `kingdee_*` 工具查涉及的科目核算维度配置
- **解决方向**:
  - 按"凭证生成报告"逐单处理:缺科目就补科目绑定、缺维度就补维度分配、借贷不平调对称条件
  - 模板里的基础资料名称**不能有尾部空格**(常见坑,最好用编码代替)
- **链到**:如果现金流量相关项目没填,看下面 §5
- **来源**:[金蝶云星空凭证生成取不到对应科目的原因](https://m.heshuyun.com/292.html) · [金蝶云星空供应链单据生成凭证错误排查](https://vip.kingdee.com/article/11882) · [凭证生成异常,取不到科目](https://m.heshuyun.com/520.html) fetched 2026-04-23

### 场景 2.2:凭证生成了但数量不全(100 张单据只出了 80 张凭证)

- **症状**:期间内有 100 张业务单据,生成凭证后只有 80 张有对应凭证
- **常见原因**:
  1. 剩下 20 张有**数据问题**(缺科目 / 金额 0 / 维度未设),被系统跳过
  2. 这 20 张单据**审核状态不对**(未审核或反审核了)
  3. 单据类型**不在凭证生成范围**(用户选了部分单据类型)
- **诊断步骤**:调**凭证生成报告**看失败清单;核对单据审核状态
- **解决方向**:按报告逐张处理
- **来源**:[金蝶云星空会计凭证为什么查询不全](https://m.heshuyun.com/507.html) fetched 2026-04-23

### 场景 2.3:应收账款与总账对不平

- **症状**:应收款管理模块的余额和总账"应收账款"科目余额差几千 / 几万
- **常见原因**:
  1. 应收单审核了但凭证没生成 / 凭证生成错了(凭证走了其他科目)
  2. **手工凭证**——用户直接录了应收科目的凭证,绕过业务单据
  3. 核销关系错乱——已核销的收款被反核销但未重新核销
  4. 外币换算差——业务日期和凭证日期汇率不同
- **诊断步骤**:
  1. 让用户走**应收 → 对账 → 应收与总账对账**,系统会按科目列差异
  2. 按差异清单逐笔定位
- **解决方向**:手工凭证的部分需要业务重新审视;核销关系错乱需要重新核销
- **来源**:[应收账款与总账对账不平问题](https://www.chinaacc.com/wenda/detail/xt/9644074) fetched 2026-04-23

## 3. 应收 / 应付账龄对不上

### 场景 3.1:账龄分析表的原币与应收汇总表不一致

- **症状**:两张报表查同一客户,原币金额数字差异
- **常见原因**:**两张表汇率算法不同**(设计差异不是 bug)
  - 应收汇总表:用**单据业务日期**的汇率
  - 账龄分析表:用**动态计算**的汇率(通常是期末汇率)
- **诊断步骤**:让用户**按本位币(CNY)对账**——两表的本位币一般能对上;原币只能接受差异
- **解决方向**:
  - 如果客户能接受:用本位币对账,原币差异归因汇率
  - 如果不能接受:开启**汇率调整**功能,期末调平
- **来源**:[金蝶云星空应收款管理参数详解](https://zhuanlan.zhihu.com/p/350191374) · [第10期 金蝶云·星空-财务云问答精选](https://vip.kingdee.com/article/294493768869824512) fetched 2026-04-23

### 场景 3.2:账龄分组和预期不符(某客户明明欠 30 天但分在 90 天组)

- **症状**:账龄分析表里,某笔应收的账龄组别和手算不一致
- **常见原因**:
  1. **账龄起算日期**口径差异——系统默认按**单据业务日期**,用户可能期望**到期日**
  2. **账龄区间自定义**了但没重算
  3. 部分核销未释放——已收款但核销关系没完成,被系统继续算账龄
- **诊断步骤**:查账龄分析表的**过滤条件 → 账龄起算方式**参数
- **解决方向**:
  - 切换起算方式(业务日期 vs 到期日)
  - 重新核销未完成的部分
  - V8 之后账龄表支持"**按到期分组**",低版本要用公式改
- **来源**:[金蝶云星空V8应收应付账龄分析表支持设置到期分组](https://www.heshuyun.com/132.html) fetched 2026-04-23

## 4. 报表数据为空 / 不全

### 场景 4.1:打开报表全部为空

- **症状**:报表里一行数据都没有,但明明有业务发生
- **常见原因**(按出现频率):
  1. **过滤方案选错**——选了一个窄范围的过滤方案(如只查某业务员 / 某客户)
  2. **期间参数错**——选了未来期间或没业务的期间
  3. **数据权限过滤**——用户被限制只能看某组织,其他组织的数据不可见
  4. **模板取数公式错**——报表模板底层 SQL / 公式被改动了
- **诊断步骤**:
  1. 让用户重置过滤方案到"默认"
  2. 核对期间
  3. 切到 admin 看能不能看到数据——能看到 = 数据权限问题
- **解决方向**:
  - 过滤方案 → 切到"默认"或新建一个全开的过滤方案
  - 权限 → admin 调整数据权限
  - 公式被改 → 找备份还原,或重新设置报表模板
- **来源**:[金蝶云星空单据列表数据查询过滤的三种方式](https://www.zjgkd.cn/?p=1832) · [金蝶云星空在原有简单账表加自定义字段](https://www.cnblogs.com/woshinige/p/18234862) fetched 2026-04-23

### 场景 4.2:报表部分数据缺失(有些客户有,有些客户没)

- **症状**:报表里看到部分客户 / 物料,但预期应该有的某些记录不显示
- **常见原因**:
  1. **过滤条件不够宽**(如"只显示有余额"勾选了)
  2. 那些缺失的记录有**数据质量问题**(如核算维度字段为空,被系统跳过)
  3. **组织隔离**——记录属于其他组织
- **诊断步骤**:放宽过滤,重置组织,看能否出来
- **解决方向**:定位到具体记录后,查其字段完整性
- **来源**:经验归纳

### 场景 4.3:报表不能重算 / 数据陈旧

- **症状**:修改了底层数据但报表还显示旧值,"重算"按钮失效
- **常见原因**:
  1. 某些报表是**快照表**,需要触发计划任务刷新
  2. 报表权限:用户没有"重算"功能权限
  3. 计算服务未启动(涉及固定资产折旧、存货成本的报表)
- **诊断步骤**:看报表是否标注"**需后台重算**";admin 权限下试重算
- **解决方向**:通过计划任务 / 后台服务触发重算
- **来源**:[金蝶云星空报表不能重算怎么操作](https://m.heshuyun.com/1625.html) fetched 2026-04-23

## 5. 现金流量表为空 / 不平

### 场景 5.1:现金流量表打开全部为 0

- **症状**:运行现金流量表,所有项目都是 0 或大量项目缺失
- **常见原因**:
  1. **凭证没指定现金流量项目**——现金流量表依赖凭证的"现金流量项目"分录,没指定就取不到数据
  2. **科目没勾"流量项目"**——现金 / 银行存款科目要勾"核算项目 → 现金流量项目"
  3. **报表模板取数公式错**——用户改了公式导致取不到
- **诊断步骤**:
  1. 让用户打开任意一张涉及现金的凭证,看"**现金流量指定**"标签是否有数据
  2. 查现金 / 银行类科目设置,看有没有勾"流量项目"
- **解决方向**:
  - 历史凭证已审核:用**凭证指定现金流量**界面批量"自动指定"
  - 未来凭证:科目上勾"流量项目",系统会自动带出
  - 另一种方法:用"**凭证模板预设现金流量项目**",在模板层配置
- **链到**:如果凭证生成本身有问题,回到 §2
- **来源**:[金蝶云星空现金流量指定方法大全](https://www.mfangyou.net/archives/506) · [凭证指定现金流量](https://help.open.kingdee.com/dokuwiki/doku.php?id=%E5%87%AD%E8%AF%81%E6%8C%87%E5%AE%9A%E7%8E%B0%E9%87%91%E6%B5%81%E9%87%8F) fetched 2026-04-23

### 场景 5.2:现金流量表不平(主表合计 ≠ 分项合计)

- **症状**:现金流量表的"**现金及现金等价物净增加额**"和各大类加总对不上
- **常见原因**:
  1. 部分凭证的现金流量指定**漏了**(系统只统计已指定的)
  2. 有的现金流量**指定到错误大类**了(如经营活动的记成投资活动)
  3. 期初现金余额和流量表的起点不一致
- **诊断步骤**:走"**凭证 → 现金流量项目分析**"看哪些凭证没指定
- **解决方向**:批量自动指定 + 错误的手工调整
- **来源**:[调整分录法编制现金流量表要点](https://www.kingdee.com/article/1529009708139032577.html) fetched 2026-04-23

## 临时话术

### 发现是口径 / 设计差异(不是 bug)

> "这个差异不是系统 bug,是**两张表的设计口径不同**——[A 表]用 X 方法算,[B 表]用 Y 方法算。你们的需求是对同一口径吗?如果是,我建议:
> - 统一看 [推荐表]
> - 或者用**本位币**对账,原币差异归因汇率差
> 如果业务需要精确一致,需要启用**汇率调整**功能(涉及月末调整凭证,让财务同事评估)。"

### 数据问题定位到具体几张单据

> "定位到了差异来自这几张单据:
> 1. [单据号 1] - [差异原因]
> 2. [单据号 2] - [差异原因]
>
> 建议处理:按 `common/systematic-debugging` 五步——先**反审核 → 改正 → 重审核**,如果涉及已关账期间需要**先反关账**(走 `standard-errors` §4.3)。
>
> 处理完后再跑一次报表对账确认。"

### 卡住(走不完排障)

> "我用只读工具查到的信息只能指向 [模糊方向],没法精确定位。建议:
> 1. 让客户在**数据对账报告**上截图(具体到差异金额 + 客户 + 物料)
> 2. 把原始单据和凭证都导出来
> 3. 如果是汇率 / 成本核算这种复杂业务,建议找客户的财务主管 + 金蝶顾问一起看
> 这类问题单靠 agent 的元数据工具是查不全的——需要真实业务判断。"
