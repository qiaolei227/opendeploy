# K/3 Cloud 性能问题诊断 🟡

<!-- 来源:help.open.kingdee.com / vip.kingdee.com / m.heshuyun.com / blog.csdn.net 公开内容综合;实证状态:🟡 常见场景索引,具体解决方案以客户环境为准 -->

覆盖"系统跑得慢 / 卡 / 超时"——单据保存慢 / 列表查询慢 / 报表跑不出 / 客户端卡死 / MRP 运算超时。

**核心纪律**:
- **不要立刻提议写插件优化**——Python 几乎不会让业务变快,**反而可能更慢**
- **绝大多数性能问题是基础设施 / SQL 索引 / 数据规模 / 配置过宽**,不是金蝶 bug
- 走 `common/systematic-debugging` 五步,定位是**前端慢 / 网络慢 / 服务端慢 / 数据库慢** 哪一段

agent 在性能问题上**最常犯的错**:用户一说"慢",立刻提议"我帮你写个插件优化"——**这是反模式**。先定位瓶颈在哪,再决定方案。

---

## 0. 通用诊断流程(每个性能场景都先走)

### 第一步:定位瓶颈在哪一段

| 段 | 用户能观察到什么 | 工具 |
|---|---|---|
| **客户端渲染** | 表单打开慢但保存快 / 仅这一台机器慢 | F12 浏览器开发者工具(Web 客户端);客户端日志 |
| **网络** | 不同机器都慢,但 ping / RDP 服务器很快 | `ping <server> -t` / `tracert` / 让用户换 VPN 通道 |
| **服务端 .NET** | 操作要 30 秒以上,服务器 CPU 拉高但 SQL Server 闲 | 服务器任务管理器看 IIS / `K3Cloud` 工作进程 CPU + 内存 |
| **SQL 数据库** | 慢操作时 SQL Server 实例 CPU / 磁盘 IO 拉高 | SQL Server 活动监视器、`sp_whoisactive` |

**让用户**(或客户 IT)**配合做这一步**——agent 的工具看不到客户端 / 服务器 / SQL Server 状态,**单凭对话很难定位**。

### 第二步:按定位结果选诊断方向

- 客户端慢 → §1.1 / §1.2
- 网络慢 → 找客户 IT,不在 agent 范围
- 服务端 .NET 慢 → §2 / §3 / §5
- SQL 慢 → §2 / §3,以及让 IT 看 SQL Server 索引

## 1. 客户端慢 / 卡死

### 场景 1.1:某台机器特别慢(其他机器正常)

- **症状**:用户 A 操作很慢,但同账号在用户 B 机器上正常
- **常见原因**:
  1. **客户端版本不一致**——A 没自动更新到最新补丁
  2. A 机器的客户端**本地缓存损坏**
  3. A 机器**杀软 / 防火墙**拦截了 K/3 Cloud 进程
  4. A 机器硬件 / 系统资源不足
- **诊断步骤**:
  1. 看 A 机器客户端**右下角版本号**和服务端管理中心的版本对比
  2. 让 A 完全退出客户端(任务管理器杀进程),清缓存:`%LOCALAPPDATA%\Kingdee\Kingdee.BOS.Configuration` 和 `%USERPROFILE%\Documents\Kingdee\K3Cloud\Cache`,重新启动
- **解决方向**:重装客户端或清缓存
- **链到**:如果重装后还慢,看 §1.2
- **来源**:[第1-8期 安装与性能环境问答精选](https://vip.kingdee.com/article/375027675112386048) · [熊说星空-星空客户端常见问题](https://vip.kingdee.com/article/255811123503793152) fetched 2026-04-23

### 场景 1.2:客户端卡死(无响应、Ctrl+Alt+Del 才能退)

- **症状**:用户操作过程中客户端突然无响应,鼠标点哪都没用
- **常见原因**:
  1. **网络连接中断**(无线网切换 / VPN 掉)
  2. **客户端要访问外部资源**(如金蝶在线服务、广告位)且**外网被防火墙拦**
  3. **服务端长事务**(用户触发了一个慢查询,客户端等响应)
  4. **客户端版本 bug**(某些版本有已知卡死缺陷)
- **诊断步骤**:
  1. 让用户先确认有线 / 无线网络是否稳定
  2. 检查防火墙是否禁了客户端访问外网(企业内网常见)
  3. 客户端日志:`%USERPROFILE%\Documents\Kingdee\K3Cloud\Log` 找 `Exception`
  4. 客户端按 **Ctrl+F12** 调出性能 / 日志面板(因版本而异),或 **Ctrl+Alt+Shift+M** 性能监控
- **解决方向**:
  - 网络问题让 IT 处理
  - 外网拦截:让 IT 给 K/3 Cloud 客户端开行
  - 长事务:让用户取消操作,看是不是某次报表 / 查询太宽
  - 已知版本 bug:升级到最新补丁
- **来源**:[第1-8期 安装与性能环境问答精选](https://vip.kingdee.com/article/375027675112386048) · [熊说星空](https://vip.kingdee.com/article/255811123503793152) · [上机操作日志](https://help.open.kingdee.com/dokuwiki/doku.php?id=%E4%B8%8A%E6%9C%BA%E6%93%8D%E4%BD%9C%E6%97%A5%E5%BF%97) fetched 2026-04-23

## 2. 单据保存 / 审核慢

### 场景 2.1:大单据明细多,保存超过 30 秒

- **症状**:销售订单明细 100 行+,保存要等很久(30s ~ 几分钟)
- **常见原因**:
  1. **业务规则 / 公式字段太多**——每行都要算,行多就慢
  2. **客户现场插件**在 `BeforeSave` 做了循环查询(N+1 问题)
  3. **核算维度联动**——每行都要查基础资料
  4. **数据库索引**缺失或失效
  5. **服务端配置不够**(IIS 应用池工作进程 / .NET GC)
- **诊断步骤**:
  1. 让用户尝试**精简一份测试单**(只 5 行),保存看是否还慢——慢 = 单据本身问题(业务规则);快 = 行数堆积问题
  2. 用 `kingdee_list_form_plugins('<baseObjectId>')` 查挂了什么插件
  3. 让客户 IT 在 SQL Server 抓**保存时刻的活动 SQL**(`sp_whoisactive`)
- **解决方向**:
  - 业务规则太多:BOS Designer 优化规则(如改异步)
  - 插件 N+1:让技术方重写插件,改批量查询
  - 索引:让客户 DBA 加索引(只读维护,不改业务表 schema)
  - 配置:让客户 IT 调 IIS 应用池 + .NET GC
- **链到**:如果是本产品生成的插件慢,走 `python-plugin-index/prompts/debugging`
- **来源**:[金蝶云星空学习笔记02](https://zhuanlan.zhihu.com/p/641790877) · [金蝶系统很卡住了点不动怎么办](https://m.heshuyun.com/347.html) fetched 2026-04-23

### 场景 2.2:F7 选基础资料(物料 / 客户)弹窗很慢

- **症状**:点 F7 弹基础资料选择框,弹出要 5 秒以上
- **常见原因**:
  1. **基础资料数据量大**(物料 10 万+,弹窗加载慢)
  2. **网络往返慢**——网卡设置 / 网络抖动(一个真实案例:IPv4 校验失败)
  3. F7 默认带的过滤条件太宽
  4. **客户端缓存未启用**或缓存损坏
- **诊断步骤**:
  1. 让用户在**搜索框输入关键字再展开**——验证是不是默认全加载导致
  2. 让 IT 检查**网卡 IPv4 校验和卸载**等高级设置(社区有真实案例)
- **解决方向**:
  - 启用 F7**懒加载**和**模糊搜索**
  - 让 IT 调网卡设置
  - 大基础资料考虑分组管理
- **来源**:[金蝶云星空网络优化案例(F7 选资料卡顿)](https://zhuanlan.zhihu.com/p/641790877) fetched 2026-04-23

## 3. 列表查询 / 过滤慢

### 场景 3.1:打开单据列表慢

- **症状**:点"销售订单"菜单进列表,要等 10 秒+ 才出现行
- **常见原因**:
  1. **默认过滤条件太宽**(默认查全部历史)
  2. **数据库表行数巨大** + **索引缺失**
  3. **数据权限过滤**复杂(多组织 + 多角色)
  4. **列表显示了过多列**(默认显示插件计算列)
- **诊断步骤**:让用户**修改默认过滤方案**为窄范围(如最近 30 天)
- **解决方向**:
  - 改默认过滤方案
  - 让 DBA 检查 `T_SAL_*` / `T_PUR_*` 等业务表的索引(只读维护,不改 schema)
  - 减少列表默认列数
- **来源**:[金蝶云星空单据列表数据查询过滤的三种方式](https://www.zjgkd.cn/?p=1832) · [星空SQL数据库配置和优化](https://blog.csdn.net/YZYKJY/article/details/126380217) fetched 2026-04-23

### 场景 3.2:复杂过滤(多条件 + 高级查询)慢

- **症状**:简单过滤快,加了几个高级条件后变慢
- **常见原因**:
  1. 高级过滤的字段**没有索引**
  2. 用了**模糊匹配 LIKE %X%**(全表扫描)
  3. **关联了大基础资料**(如关联客户分类)
- **解决方向**:
  - 让 IT 加索引(具体字段交给 DBA 决策)
  - 改前缀匹配 LIKE 'X%'(走索引)
  - 减少关联条件
- **来源**:[金蝶云星空SQL数据库配置和优化](https://blog.csdn.net/YZYKJY/article/details/126380217) fetched 2026-04-23

## 4. 报表跑不出 / 超时

### 场景 4.1:大报表点"查询"后转圈很久或直接超时

- **症状**:点报表查询按钮,等几分钟后报"请求超时"或浏览器假死
- **常见原因**:
  1. **期间太大**(查 1 整年的明细)
  2. **数据范围太宽**(全公司全产品,没收窄过滤)
  3. **报表底层 SQL 复杂**(多表关联 + 多层子查询)
  4. **服务端 / SQL 资源紧张**
- **诊断步骤**:让用户**先收窄一个维度**(改成 1 个月 / 1 个组织),看能不能跑出来
- **解决方向**:
  - 收窄维度,分批查
  - 大报表跑**计划任务夜间预跑**,白天看快照
  - 让 IT 评估服务器资源
  - 复杂报表考虑**专门的报表平台 BI**(K/3 Cloud 不擅长大数据分析)
- **来源**:[金蝶云星空报表不能重算怎么操作](https://m.heshuyun.com/1625.html) fetched 2026-04-23

### 场景 4.2:某些报表永远跑不完(其他正常)

- **症状**:特定一张报表(如成本明细 / 应收应付明细)永远超时,其他报表正常
- **常见原因**:
  1. 这张报表的**底层取数公式**有性能问题
  2. **缺少专用索引**支持这张报表
  3. **历史数据太多**(几年的数据没归档)
- **诊断步骤**:让 DBA 抓这张报表执行时的 SQL,看执行计划
- **解决方向**:加索引 / 数据归档 / 改报表实现
- **来源**:[星空SQL数据库配置和优化](https://blog.csdn.net/YZYKJY/article/details/126380217) fetched 2026-04-23

## 5. MRP / 复杂运算超时

### 场景 5.1:MRP 运算几小时跑不完或超时退出

- **症状**:点"MRP 运算"后等很久,提示超时或服务端断开
- **常见原因**:
  1. **计划方案太宽**——所有物料 / 所有组织 / 全期间一次跑
  2. **物料 BOM 层级很深**(分解层数多)
  3. **历史数据没清理**——MRP 拿历史在算
  4. **服务端资源不足**(MRP 是计算密集任务)
- **诊断步骤**:让用户看 MRP 方案的过滤范围
- **解决方向**:
  - **缩小方案范围**——按物料类 / 按组织拆,分批跑
  - 设置**不释放预留**的方案 + **定时调度**夜间跑(社区推荐)
  - 历史数据**定期归档**清理
  - 服务器加资源
- **链到**:如果计划方案的复杂度超过工具能调,告诉用户找金蝶顾问做生产计划优化方案
- **来源**:[金蝶云·星空 MRP 运算日常运维及常见问题](https://vip.kingdee.com/article/273206140803533824) · [金蝶云星空软件生产计划教程](https://www.jiandaoyun.com/blog/article/405819/) fetched 2026-04-23

## 6. 数据库 / 服务端基础设施(让用户找 IT)

### 场景 6.1:整个系统所有人都慢

- **症状**:不同机器 / 不同账号 / 不同操作都慢
- **常见原因**(这块超出 agent 能直接处理的范围,**让客户 IT 检查**):
  1. **SQL Server 内存不足**——金蝶推荐预留 2GB 给操作系统,其余给 SQL Server 实例
  2. **应用池配置不足**——IIS 应用池工作进程数不够,GC 不优化
  3. **磁盘 IO 瓶颈**——日志盘 / 数据盘混用
  4. **网络抖动 / 带宽不够**
  5. **病毒扫描 / 杀软干扰**——杀软扫 K/3 Cloud 进程
  6. **死锁 / 长事务堆积**
- **诊断步骤**:让 IT 检查:
  - SQL Server 配置(`sp_configure`,看 `max server memory` 等)
  - SQL Server 启用"**即席工作负荷优化**"(`Optimize for Ad hoc Workloads`)
  - **锁定内存页**组策略
  - 服务器资源监控(CPU / 内存 / 磁盘 IO)
  - 网卡设置(IPv4 校验和卸载等高级)
- **解决方向**:让 IT 按金蝶官方[**性能优化白皮书**](https://m.heshuyun.com/998.html)逐项过
- **来源**:[星空SQL数据库配置和优化](https://blog.csdn.net/YZYKJY/article/details/126380217) · [金蝶云星空学习笔记02](https://zhuanlan.zhihu.com/p/641790877) · [金蝶系统很卡住了点不动怎么办](https://m.heshuyun.com/347.html) fetched 2026-04-23

## 临时话术

### 性能问题第一时间(不要立刻提优化方案)

> "在我提优化方案前,需要先**定位瓶颈**——按 `common/systematic-debugging` 第 1 步'症状钉死':
> 1. 是**所有人**都慢还是**某些人**慢?
> 2. 是**所有操作**都慢还是**特定操作**(某张单据 / 某个报表)?
> 3. 慢操作时**服务器 CPU / SQL Server CPU / 网络带宽**有没有拉高?
>
> 这些信息客户 IT 配合就能拿到。**不要先猜原因**——同样的'慢',索引问题 / 网络问题 / 插件问题 / 配置问题,解法完全不一样。"

### 定位到客户基础设施问题(超出本产品范围)

> "排查后定位是**基础设施层**问题——具体是 [SQL 内存配置 / 网卡设置 / 应用池配置 / 索引缺失]。这块超出本产品(实施交付智能体)能直接修的范围,需要客户 IT / DBA 处理。
>
> 给你一份金蝶官方建议的**对照清单**:
> 1. SQL Server 设 `max server memory`(预留 2GB 给 OS,剩余给 SQL)
> 2. 启用"即席工作负荷优化"(`Optimize for Ad hoc Workloads`)
> 3. 启用"锁定内存页"组策略(给 SQL Server 用户)
> 4. 检查网卡:禁用 IPv4 / IPv6 校验和卸载(社区案例)
> 5. 检查 IIS 应用池工作进程数 / 回收策略
>
> 让 IT 逐项过。如果客户没有专业 DBA,可以联系金蝶官方的**性能优化服务**(付费)。"

### 用户一上来就说"帮我写插件优化"(纠正反模式)

> "等等——'慢'的根因还没定位,**写插件不是优化方案**,**Python 插件本身可能让业务更慢**(每次都解释执行)。
>
> 按正确顺序:
> 1. 先排查**配置 / 索引 / 网络 / 基础设施**(`performance-issues` §6)
> 2. 再看**业务规则 / 数据规模**(§2 / §3)
> 3. 实在改不了再考虑**改业务流程**(分批 / 异步)
>
> 90% 的'慢' 在第 1-2 步就解决了。先把基础查一遍,我陪你走。"
