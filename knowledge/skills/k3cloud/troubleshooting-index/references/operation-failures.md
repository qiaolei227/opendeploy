# K/3 Cloud 操作失败诊断 🟡

<!-- 来源:help.open.kingdee.com / vip.kingdee.com / m.heshuyun.com / blog.csdn.net 公开内容综合;实证状态:🟡 常见场景索引,具体解决方案以客户环境为准 -->

覆盖用户"想做一个操作但系统拒绝"的场景——下推 / 审核 / 反审核 / 删除 / 保存被拦。

**和 standard-errors 的区别**:那个是系统跑不起来 / 流程走不下去(登录 / 关账 / 启用);本文件是单据层面的操作被拒绝。

**核心纪律**:操作失败**绝大多数是业务逻辑拦截**(金蝶原厂规则或客户二开规则),不是 bug。agent 姿势:

1. **先看报错原文**——金蝶的报错文本往往直接说了原因(如"已下推下游单据"),不要无视
2. **判断拦截来自哪一层**——金蝶标准 / BOS 业务规则 / Python-DLL 插件 / 数据权限
3. **按层对症**——标准规则改配置,二开规则看代码

走 `common/systematic-debugging` 五步流程,不要一看到"失败"就猜。

## 1. 单据下推失败

### 场景 1.1:点"下推"按钮弹"没有可下推的数据"

- **症状**:销售订单 → 发货通知单 / 采购订单 → 采购入库,点下推后提示"没有符合条件的数据 / 不满足推数条件"
- **常见原因**(按出现频率):
  1. **转换规则未启用**或标了"运行时隐藏"
  2. **推数启动条件不满足**——源单状态不对(没审核 / 已关闭)
  3. **数量已推完**——剩余可推数为 0
  4. **字段映射关系断了**——必填字段在目标单据找不到源字段
  5. **转换插件拦了**——客户现场在转换规则上挂了 Python / DLL 插件,插件内条件拒绝了
- **诊断步骤**:
  1. 让用户打开**BOS Designer → 单据转换**,看这个转换规则的**启用状态**和**启动条件**
  2. 查**剩余可推数量**——销售订单列表通常有"未推数量"列
  3. 看报错文本里有没有中文业务描述(客户自定义的)——**有就是二开插件拦的**
- **解决方向**:
  - 转换规则没启用 → BOS Designer 启用
  - 源单状态 / 数量问题 → 让用户核对源单业务状态
  - 如果是二开插件拦 → 让用户把插件代码贴出来(通常在扩展对象上,可用 `kingdee_list_form_plugins` 反查)或找技术方定位
- **链到**:如果是客户现场二开插件问题,涉及的是客户自有代码,不在本产品排障范围
- **来源**:[单据转换常见问题-下推操作选择不到转换规则](https://vip.kingdee.com/article/579637177575611392) · [金蝶云单据下推,转换规则](https://blog.csdn.net/caibiyige1234/article/details/128170115) · [销售管理常见WEBAPI问题及其他常见问题汇总指导](https://vip.kingdee.com/article/208667621993517568) fetched 2026-04-23

### 场景 1.2:下推的目标单字段值不对

- **症状**:下推成功但目标单据的某字段值**不是**预期的(数量少了 / 客户变了 / 仓库错了)
- **常见原因**:
  1. **转换规则的字段映射错了**(如源单的"客户"映射到了目标单的"供应商")
  2. **公式字段**算法错
  3. **默认取值**覆盖了源单值
  4. 客户现场的**转换插件**在映射后又改了值
- **诊断步骤**:
  1. BOS Designer → 单据转换 → 字段映射列表,逐字段核对
  2. 看是否有"**字段初始值 / 默认值**"优先于源单值
  3. 看扩展对象上有没有转换类插件
- **解决方向**:
  - 标准映射问题 → BOS Designer 修映射
  - 二开插件导致 → 让客户技术方审查
- **链到**:BOS 转换规则本身的配置看 `bos-features-index/references/convert-rules`(如果存在)
- **来源**:[金蝶云星空单据转换设置](https://blog.csdn.net/leo0769/article/details/127496489) fetched 2026-04-23

## 2. 单据审核失败

### 场景 2.1:点"审核"报"字段必填"但字段已填

- **症状**:客户说"客户 / 物料 / 仓库字段明明填了,审核时还是报'XX 必填'"
- **常见原因**:
  1. **字段值是空字符串**(看起来有但实际是空格或特殊字符)
  2. **字段是基础资料**,基础资料本身被禁用或未分配到当前业务组织
  3. **BOS 业务规则 / 保存校验规则**拦了——金蝶允许配"条件必填"(基于其他字段判断)
  4. **审核类插件拦了**——在 `BeforeAudit` / `AuditValidation` 事件里抛了 `KDException`
- **诊断步骤**:
  1. 让用户把字段**清空再重填**一次(基础资料用 F7 重选)
  2. 看 BOS Designer 上这个单据的**业务规则**列表
  3. 用 `kingdee_list_form_plugins('<baseObjectId>')` 查挂了哪些插件
- **解决方向**:
  - 重选基础资料 = 值变干净
  - 如果 BOS 规则拦:让技术方审查规则配置
  - 如果插件拦:让技术方审查插件代码
- **来源**:[金蝶云星空业务单据审核校验根据条件分录数值必录](https://www.cnblogs.com/lanrenka/p/17792672.html) · [金蝶云星空表单服务规则设置-基础资料和复选框判断](https://blog.csdn.net/qq_33881408/article/details/134878374) fetched 2026-04-23

### 场景 2.2:审核报"权限不足 / 该用户不是审核人"

- **症状**:点审核后提示权限不足
- **常见原因**:
  1. 用户**角色没授权**该单据的审核功能
  2. 启用了**审批流**,该用户不在审批节点
  3. 启用了**专岗作业**(某些单据要求录入员 ≠ 审核员)
  4. 数据权限过滤——该用户能看到但不能审核特定范围的数据
- **诊断步骤**:admin 试审核,对比权限差异
- **解决方向**:admin 授权;或审批流节点调整
- **来源**:[金蝶云星空-当前用户不是审核人](https://blog.csdn.net/qq_25905177/article/details/136036128) · [总账记账凭证反审核的前提和流程](https://vip.kingdee.com/questions/148207/answers/174933) fetched 2026-04-23

## 3. 反审核失败

### 场景 3.1:反审核报"存在下游业务"

- **症状**:销售订单想反审核,提示"已下推下游单据,需先删除下游再反审核"
- **常见原因**:原单据已下推到下游(如销售订单 → 发货通知单),下游存在时不允许反审核
- **诊断步骤**:用 `kingdee_*` 工具或直接让用户在单据详情页查**关联关系**(通常有"追溯"按钮)
- **解决方向**:
  - **自底向上处理**:先反审核 / 删除下游,再反审核上游
  - 如果下游已经过账 / 核销,要更深层处理(反过账 → 反核销 → 反审核 → 删除)
- **链到**:如果凭证已生成,看 §3.2
- **来源**:[金蝶云星空反审核前提条件介绍](https://m.heshuyun.com/1560.html) · [第1-15期 供应链云问答精选](https://vip.kingdee.com/article/297868445403148544) fetched 2026-04-23

### 场景 3.2:反审核报"已生成凭证"

- **症状**:财务单据想反审核,提示"已生成凭证,请先删除凭证"
- **诊断步骤**:让用户走"凭证 → 查询" 找到对应凭证
- **解决方向**:
  1. 先**凭证反过账**(如果已过账)
  2. 再**凭证反审核**(如果已审核)
  3. 再**删除凭证**
  4. 再反审核原单据
- **来源**:[金蝶云星空反审核前提条件介绍](https://m.heshuyun.com/1560.html) fetched 2026-04-23

### 场景 3.3:反审核报"已核销 / 已结算"

- **症状**:应收 / 应付单想反审核,提示"已核销,请先反核销"
- **诊断步骤**:让用户查**核销记录**——"业务查询 → 核销记录"
- **解决方向**:先反核销 → 再反审核
- **来源**:[金蝶云星空反审核前提条件介绍](https://m.heshuyun.com/1560.html) fetched 2026-04-23

### 场景 3.4:自动生成的单据无法反审核(应收应付结算单、费用报销后自动生成的应付单等)

- **症状**:这类单据没有"反审核"按钮或点了报错
- **常见原因**:**系统自动生成的单据**(结算 / 核销 / 报销产生的应付)**不允许直接反审核**
- **解决方向**:必须反向操作**源头**(反结算 / 反核销 / 反报销),自动生成的单据会被系统级联反处理
- **来源**:[第1-15期 供应链云问答精选](https://vip.kingdee.com/article/297868445403148544) · [金蝶云星空应用常见问题集锦——财务篇](https://mindee.cn/index.php?id=2987) fetched 2026-04-23

## 4. 删除失败

### 场景 4.1:删除单据报"被引用"

- **症状**:想删一张单据,提示"该单据已被下游引用,不能删除"
- **常见原因**:单据已下推,下游引用它作为源单
- **诊断步骤**:同 §3.1,查关联关系
- **解决方向**:**先处理下游,再删上游**
- **来源**:[金蝶云社区综合](https://vip.kingdee.com/article/45669) fetched 2026-04-23

### 场景 4.2:删除报"期间已关账"

- **症状**:提示"当前期间已关账,不允许删除"
- **诊断步骤**:看单据的**业务日期**所在期间是否关账
- **解决方向**:
  - 如果必须删:**先反关账**(走 `standard-errors` §4.3),再删,再重新关账
  - 如果业务日期可以改:反审核 → 改业务日期到未关期间 → 重审核 / 删除
- **链到**:反关账流程看 `standard-errors` §4.3
- **来源**:[进销存单据日期不能小于系统结账日期](https://vip.kingdee.com/article/47741212045148212) fetched 2026-04-23

### 场景 4.3:基础资料(客户 / 物料)删除失败

- **症状**:想删一个客户或物料,报"存在业务发生,不允许删除"
- **常见原因**:该基础资料已经被业务单据引用(哪怕是一张很早的单据)
- **解决方向**:
  - 基础资料**通常只禁用不删除**——改"使用状态"为"禁用"即可
  - 真要删除:必须**先删除所有引用它的业务单据**(不现实)
  - 实施建议:直接禁用,新建一个正确的替代
- **来源**:经验归纳 + 金蝶官方推荐

## 5. 保存失败

### 场景 5.1:保存报"字段必填 / 字段格式错"(HTTP 500 + 字段必填的 WebAPI 陷阱)

- **症状**:明明字段都填了,保存时报"XX 字段必填"
- **常见原因**:
  1. **字段顺序问题**——金蝶对某些字段有顺序要求,乱序会导致自动化逻辑覆盖成空(尤其通过 WebAPI 保存时)
  2. **基础资料未分配到当前业务组织**——字段看起来填了,但系统发现基础资料不在当前组织范围,视为无效
  3. **BOS 字段依赖**——某字段的"必填"是条件依赖(选了 X 后 Y 变必填),但用户没注意
  4. 客户**扩展字段**没给 label / 没分配到对应组织
- **诊断步骤**:
  1. 界面保存:F5 刷新表单重填
  2. WebAPI 保存:调 API 时加 `IsAutoAdjustField: "true"` 或把关键字段(客户 / 组织 / 币别)放 `Model` 最前
  3. 查基础资料的组织分配
- **解决方向**:
  - 字段顺序:调整 JSON 顺序或加 `IsAutoAdjustField`
  - 基础资料组织:admin 在基础资料上重新分配组织
- **链到**:如果是 WebAPI 问题,看 `webapi-index/references/error-handling`
- **来源**:[对接金蝶 Open API 500 字段必填](https://blog.csdn.net/Monarchess_1234/article/details/122563222) · [金蝶云 WebAPI 开发权威文档](https://blog.csdn.net/weixin_42453228/article/details/152535811) fetched 2026-04-23

### 场景 5.2:保存报"唯一约束 / 单号重复"

- **症状**:报"FBILLNO 重复 / 唯一键冲突"
- **常见原因**:
  1. 单号规则和实际使用的单号序列失调
  2. 并发保存(两个用户几乎同时保存)
  3. 历史导入数据把某些单号占了
- **诊断步骤**:让用户查**单号规则**是否正确;是否有手工指定单号的情形
- **解决方向**:单号规则修复;并发问题重试即可
- **来源**:经验归纳

### 场景 5.3:保存报"插件异常"

- **症状**:保存时报错,信息看起来不像标准提示(通常含中文业务描述或开发商前缀 `F_XXXX_`)
- **常见原因**:客户现场 Python / DLL 插件在 `BeforeSave` 抛了异常
- **诊断步骤**:用 `kingdee_list_form_plugins('<baseObjectId>')` 查挂了哪些插件;让用户提供完整报错文本和客户端日志
- **解决方向**:这是**客户现场代码**问题,不是产品能独立解决的。让客户技术方审查插件代码
- **链到**:如果是**本产品生成的**插件报错,走 `python-plugin-index/prompts/debugging` 五步
- **来源**:经验归纳 + 实证经验

## 临时话术

### 发现是业务规则 / 审批流拦截(不是 bug)

> "这个'失败'不是 bug,是**金蝶的业务规则**——[规则名]要求 [具体条件]。你有两条路:
> 1. **按规则走**:先 [前置动作],再重试
> 2. **关闭 / 调整规则**(如果实施确实不合理):让 admin 进 BOS Designer 修改配置
> 推荐第 1 条,规则通常有业务合理性,不要轻易关。"

### 发现是客户现场二开干扰

> "报错文本里有中文业务描述——看起来像**你们现场的 Python / DLL 插件**拦的。我用 `kingdee_list_form_plugins` 查了 [baseObjectId],挂了这些插件:[列出来]。这不是本产品能排障的代码——建议:
> 1. 找你们当初做二开的团队定位,把**完整报错文本**和**客户端日志**给他们
> 2. 临时 workaround:如果业务不急,让他们把插件临时禁掉重试,确认是否插件导致
> 如果需要我帮你看插件代码逻辑,可以把 `.py` 文件内容发给我(注意脱敏客户数据)。"

### 流水线反处理(下游拦了上游)

> "定位到了:[原单据]已经下推到了 [下游单据],所以不能直接反审核 / 删除。**必须从最下游往上逐步反处理**:
> 1. 反过账 [涉及的凭证](如果有)
> 2. 反审核 [凭证](如果有)
> 3. 删除 [凭证](如果有)
> 4. 反核销 [核销记录](如果有)
> 5. 反审核 → 删除 [下游单据]
> 6. 反审核 [原单据]
>
> 每一步做完都看系统是否放行——按 `common/systematic-debugging` 第 3 步'一次验一个假设'执行。"
