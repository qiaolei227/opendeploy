# 固定资产(FA)标准功能字典

> FA(Fixed Asset)模块以**资产卡片**为核心,贯穿"申请 → 入账 → 折旧 → 变更 / 转移 / 借用 → 盘点 → 处置 / 减值 → 期末结账"全生命周期。
>
> 实施顾问最常踩的坑:把"折旧方法切换"、"按部门分摊折旧"、"资产标签打印"、"在建工程转固"这些**会计政策 + 标准单据**就能解决的需求误判成 BOS 二开。

---

## 模块菜单根

K/3 Cloud 标准菜单:**资产管理云 → 固定资产**(下含:基础设置 / 启用与初始化 / 日常管理 / 折旧管理 / 期末处理 / 报表)。
所有路径以"资产管理 → 固定资产"为前缀。

## 1. 资产卡片

### 资产卡片(新增)
- **需求关键词**:"录入资产"、"资产入账"、"建立资产档案"
- **标准功能**:**资产卡片** 是 FA 的核心单据,记录原值 / 累计折旧 / 减值准备 / 折旧方法 / 折旧年限 / 净残值 / 使用部门 / 使用人 / 存放位置 / 资产类别。可手工新建,也可从**资产采购订单 / 资产验收单**下推生成
- **启用路径**:资产管理 → 固定资产 → 日常管理 → 资产卡片(新增)
- **典型误判**:顾问写脚本"采购入库后自动建卡片" — **标准下推链已经覆盖**,不需要二开;真要的是配置好"资产采购订单 → 资产卡片"的下推规则
- **OpenDeploy 工具覆盖**:❌(未对资产卡片单据做专门 helper,通用 `kingdee_*` 元数据工具可读 schema)
- **来源**:[固定资产使用流程](https://help.open.kingdee.com/dokuwiki/doku.php?id=%E5%9B%BA%E5%AE%9A%E8%B5%84%E4%BA%A7%E4%BD%BF%E7%94%A8%E6%B5%81%E7%A8%8B) · fetched 2026-04-23

### 初始化卡片
- **需求关键词**:"启用前历史资产怎么进系统"、"老资产数据迁移"
- **标准功能**:**初始化卡片** 单据,专用于启用前的历史资产录入,允许直接填累计折旧 / 已使用月数,**不会**生成入账凭证(因为属于初始化数据)
- **启用路径**:资产管理 → 固定资产 → 启用与初始化 → 初始化卡片
- **典型误判**:顾问拿 Excel 用 INSERT 灌 `T_FA_CARD` — **走初始化卡片单据**,系统帮你处理累计折旧追溯 + 期间归属

### 资产变更单
- **需求关键词**:"修改资产部门"、"调整折旧年限"、"原值调整"、"使用人变更"
- **标准功能**:**资产变更单** 用于卡片审核后的留痕修改。可改:**财务**(原值 / 累计折旧 / 累计减值准备 / 净残值 / 折旧方法 / 折旧年限 / 工作量明细)、**实物**(状态 / 位置 / 规格型号)、**分配**(部门 / 使用人 / 分摊比例,合计必须 100%)、**附属设备**
- **启用路径**:资产管理 → 固定资产 → 日常管理 → 资产变更单
- **典型误判**:顾问写 Python 在 AfterSave 改 `T_FA_CARD` — **标准变更单已覆盖且有审批留痕**;直接改卡片表会绕过折旧重算逻辑

## 2. 折旧方法 与 会计政策

### 会计政策(折旧规则中心)
- **需求关键词**:"折旧规则在哪配"、"统一折旧方法"
- **标准功能**:**会计政策** 在"资产政策"页签按**资产类别**配置:折旧方法 / 折旧年限 / 残值率(或残值额) / 折旧政策(清理当期是否计提)。新建卡片时按类别**自动带出**默认值,卡片级允许单张覆盖
- **启用路径**:资产管理 → 固定资产 → 基础设置 → 会计政策(资产政策页签)
- **典型误判**:顾问写脚本"卡片保存时按类别强制刷折旧方法" — **新建卡片本来就自动带**,刷的是已审核卡片的话应该走资产变更单
- **重要限制**:会计政策一经审核使用不能随意变更,只允许在"资产政策"页签**追加新资产类别**,已有数据禁止删除
- **OpenDeploy 工具覆盖**:❌
- **来源**:[会计政策](https://help.open.kingdee.com/dokuwiki/doku.php?id=%E4%BC%9A%E8%AE%A1%E6%94%BF%E7%AD%96) · fetched 2026-04-23

### 平均年限法
- **需求关键词**:"平均折旧"、"直线法"、"按月平均摊销"
- **标准功能**:**平均年限法**(直线法)是默认折旧方法。系统提供两种公式:**以原值为起点**(静态)+ **以净值为起点**(动态)
- **启用路径**:基础设置 → 折旧方法(选择"平均年限法"+ 公式类型)→ 会计政策中按类别引用
- **典型误判**:顾问以为"动态法"要二开 — 标准就有,选公式类型即可
- **OpenDeploy 工具覆盖**:❌
- **来源**:[计提折旧](https://help.open.kingdee.com/dokuwiki_std/doku.php?id=%E8%AE%A1%E6%8F%90%E6%8A%98%E6%97%A7) · fetched 2026-04-23

### 工作量法
- **需求关键词**:"按行驶里程折旧"、"按生产小时折旧"、"运输车 / 机床折旧"
- **标准功能**:**工作量法** 按"工作量管理"录入的当期工作量(里程 / 小时 / 件数)分摊原值 - 净残值。需在卡片上录工作总量 + 单位
- **启用路径**:基础设置 → 折旧方法(工作量法)+ 折旧管理 → 工作量管理(按月录入实际工作量)
- **典型误判**:顾问写 Python 算"按里程折旧" — **标准工作量法 + 工作量管理单据**直接覆盖
- **OpenDeploy 工具覆盖**:❌
- **来源**:[计提折旧](https://help.open.kingdee.com/dokuwiki_std/doku.php?id=%E8%AE%A1%E6%8F%90%E6%8A%98%E6%97%A7) · fetched 2026-04-23

### 加速折旧:年数总和法
- **需求关键词**:"年数总和"、"加速折旧"、"前期多提"
- **标准功能**:**年数总和法** 内置,会计政策按类别配置即可
- **启用路径**:基础设置 → 折旧方法(年数总和法)
- **OpenDeploy 工具覆盖**:❌

### 加速折旧:双倍余额递减法
- **需求关键词**:"双倍余额递减"、"DDB"、"最后两年改直线"
- **标准功能**:**双倍余额递减法** 内置。注意:DDB 在剩余 2 年自动切回平均法是会计准则要求,K/3 Cloud 内置实现不需二开
- **启用路径**:基础设置 → 折旧方法(双倍余额递减法)
- **典型误判**:顾问以为"最后 2 年切平均法"要写 Python — **标准已经按规范实现**

### 自定义折旧方法
- **需求关键词**:"我们公司有特殊折旧公式"、"按使用强度的非标算法"
- **标准功能**:固定资产参数中有 **"启用自定义折旧"** 开关,允许通过插件挂自定义算法 — 这是**官方留的二开口子**,不算误判
- **启用路径**:固定资产参数设置 → 启用自定义折旧
- **典型误判**:**反向**——这才是真正需要二开的少数场景之一,但要先确认 4 种内置法真的都不能满足
- **OpenDeploy 工具覆盖**:❌(未来可能做)

## 3. 月末计提折旧

### 计提折旧
- **需求关键词**:"月底跑折旧"、"批量折旧"、"月末折旧"
- **标准功能**:**计提折旧** 操作。流程:选择货主组织 + 会计政策记录 → 单击"计提折旧"→ 系统按符合条件的所有卡片自动计提 → "查看明细"联查每张卡的折旧数据
- **启用路径**:资产管理 → 固定资产 → 折旧管理 → 计提折旧
- **生成的单据**:折旧明细数据(可追溯,但**不直接生成凭证**——凭证由凭证模板在月末生成)
- **典型误判**:顾问问"折旧能不能按部门拆分到不同科目" — **会计政策 + 部门核算 + 凭证模板**就能做到,不是二开
- **反计提**:支持**反计提折旧**(同菜单)用于月末调整
- **OpenDeploy 工具覆盖**:❌
- **来源**:[计提折旧](https://help.open.kingdee.com/dokuwiki_std/doku.php?id=%E8%AE%A1%E6%8F%90%E6%8A%98%E6%97%A7) · fetched 2026-04-23

### 折旧调整单
- **需求关键词**:"少提了 / 多提了折旧怎么补"、"折旧手工修正"
- **标准功能**:**折旧调整单** 在不反计提的情况下手工调整某些卡片的当期折旧额
- **启用路径**:资产管理 → 固定资产 → 折旧管理 → 折旧调整单

## 4. 资产盘点

### 盘点方案 → 盘点表 → 盘盈盘亏
- **需求关键词**:"盘资产"、"年度资产盘点"、"扫码盘点"
- **标准功能**:**3 级盘点流程**——
  1. **盘点方案**:定义资产组织 / 货主组织 / 资产类别 / 卡片编码范围 / 使用部门 / 资产位置(前置:卡片已审核)
  2. **资产盘点表**:方案审核后系统按范围自动生成
  3. **盘盈盘亏单**:录入盘点结果后自动生成处理盈亏
- **启用路径**:资产管理 → 固定资产 → 期末处理 → 固定资产盘点方案 → 资产盘点表
- **典型误判**:顾问要写"盘点 APP 直连数据库" — 应该用标准盘点表 + 移动端拓展(如有),不是直写 DB
- **OpenDeploy 工具覆盖**:❌
- **来源**:[固定资产盘点方案](https://help.open.kingdee.com/dokuwiki/doku.php?id=%E5%9B%BA%E5%AE%9A%E8%B5%84%E4%BA%A7%E7%9B%98%E7%82%B9%E6%96%B9%E6%A1%88) · [资产盘点表](https://help.open.kingdee.com/dokuwiki/doku.php?id=%E8%B5%84%E4%BA%A7%E7%9B%98%E7%82%B9%E8%A1%A8) · fetched 2026-04-23

## 5. 资产清理 / 处置

### 资产处置单
- **需求关键词**:"资产报废"、"处置变卖"、"卖掉旧设备"、"部分处置"
- **标准功能**:**资产处置单** 支持**全部处置**(卡片"完全处置",不能再做其他业务)和**部分处置**(资产数量减少,剩余继续可用)。处置当期是否计提折旧由**会计政策的折旧政策**决定("清理当期计提" vs "清理当期不提")。**反审核**会清除"完全处置"标记,然后才能删除
- **启用路径**:资产管理 → 固定资产 → 日常管理 → 资产处置单
- **典型误判**:顾问以为"部分处置"要二开 — **标准支持**,在卡片明细行手工改处置数量 / 处置原值即可

## 6. 资产减值

### 累计减值准备(走资产变更单)
- **需求关键词**:"计提减值准备"、"资产减值"、"账面价值调整"
- **标准功能**:K/3 Cloud **没有独立的"减值准备单"**;通过**资产变更单**修改卡片的"累计减值准备"字段实现。**资产负债分析**报表里的固定资产期末余额 = 固定资产 - 累计折旧 - 固定资产减值准备
- **启用路径**:资产管理 → 固定资产 → 日常管理 → 资产变更单(选"累计减值准备"字段)
- **典型误判**:顾问以为要做"减值测试单据" — **标准只提供数值字段**,减值测试逻辑(可收回金额测算)不在 FA 内,要么手工算 + 录入,要么走总账单独凭证
- **OpenDeploy 工具覆盖**:❌
- **来源**:[资产变更单](https://help.open.kingdee.com/dokuwiki/doku.php?id=%E8%B5%84%E4%BA%A7%E5%8F%98%E6%9B%B4%E5%8D%95) · [资产负债分析](https://help.open.kingdee.com/dokuwiki_std/doku.php?id=%E8%B5%84%E4%BA%A7%E8%B4%9F%E5%80%BA%E5%88%86%E6%9E%90) · fetched 2026-04-23

## 7. 资本化 / 在建工程转固定资产

### 在建工程转固
- **需求关键词**:"在建工程转资产"、"完工转固"、"CIP 转固"、"工程结转"
- **标准功能**:K/3 Cloud **公开手册中未将"在建工程转固"作为 FA 的独立单据**(最新探查未找到专属页面)。常见实现:
  - **方式 A**:总账模块按"在建工程"科目余额结转,FA 侧通过**资产卡片(新增)** 手工建卡 + 引用在建工程的累计成本作为原值

## 8. 期末结账

### 固定资产结账
- **需求关键词**:"FA 月底关账"、"固定资产月结"
- **标准功能**:**固定资产结账** 操作。**前置条件**:① 必须先**计提折旧**(系统自动校验)② 所有 FA 单据**已审核** ③ 建议先**生成总账凭证 + 完成对账**。支持**反结账**
- **启用路径**:资产管理 → 固定资产 → 期末处理 → 固定资产结账
- **典型误判**:顾问写脚本"自动结账" — 结账有强校验链(折旧 → 凭证 → 对账),自动化要么撞校验,要么绕过校验留数据隐患;**应该解决"为什么没及时结账"的流程问题**而不是绕过
- **关键参数**:固定资产参数设置 → "固定资产生成的凭证未过账不允许结账"(默认开)

## 临时话术

> "你说的需求在 K/3 Cloud 标准 FA 模块里大概率是 [资产卡片 / 资产变更单 / 计提折旧 / 资产处置单 / 盘点方案] 之一覆盖的,先别上 BOS 二开。

> 我需要确认 3 件事:
> 1. **业务动作触发点**:用户说"我要做 X"时,他实际在哪个菜单 / 单据上操作?
> 2. **现有标准做法**:你客户当前是手工跑还是已经在用某个标准单据?
> 3. **真正的痛点**:是字段不够 / 校验不对 / 流程跨模块,还是只是"用户不知道有这个功能"?
>
> 把这 3 个问题的答案给我,我再判断:① 标准配置就解决 / ② 凭证模板调整 / ③ 必须 BOS 扩展 + Python 插件。"
