## 1. 执行摘要

**OpenDeploy** 是**开源**的金蝶云星空**实施交付智能体**。面向乙方实施顾问，通过AI驱动需求理解→方案生成→代码生成→BOS配置自动化，让一个顾问能完成原本必须找专业二开/外包才能做的复杂需求（L1-L4，接口对接/流程定制/模块开发）。

**核心架构哲学**（参考 Claude Code）：**Harness + Knowledge**

- **Harness**：Agent Loop框架、工具箱、UI 由产品提供
- **Knowledge**：金蝶领域深度知识库由产品提供（开源部分）+ 用户BOS运行时元数据
- **LLM**：**用户自备**（Claude / GPT / DeepSeek / Qwen / Ollama 任选）
- **服务器**：**无自营服务器**（知识库托管在GitHub，LLM在用户自己的账号）

**战略路径**：
1. 战略性就业金蝶生态公司（ISV/实施商/制造业甲方）+ 副业开发OpenDeploy
2. 开源核心产品（MIT/Apache），社区驱动流量（对标 Cline/Continue/OpenDevin）
3. 未来企业版（深度知识库+团队面板+私有部署）作为商业化路径
4. 6个月验证节奏，成功独立化、失败积累AI+ERP复合经验

**品牌矩阵**：
```
乔磊的 Open Stack
├─ OpenForge    低代码应用开发平台（另一副业）
└─ OpenDeploy   ERP实施交付智能体（本项目）
```

**语言支持**：中英双语（zh-CN主战场 + en-US国际化），从MVP-0.1起支持。

### 4.2 数据安全原则（硬红线）

**核心承诺（比业界更强）**：**产品根本不碰客户业务数据**。

这是基于一个关键认识：**AI生成代码只需要"元数据"（结构定义），不需要"业务数据"（真实内容）**。

- **元数据**（对象、字段、事件枚举、API规范）：结构信息，金蝶标准部分产品内置，客户定制部分由顾问手动告知或读取客户BOS元数据
- **业务数据**（单据、客户、金额、库存数量）：**产品从不访问**；AI生成的代码部署到BOS后，在客户ERP服务器内部访问，与我们的产品完全脱钩

**具体红线**：
- MVP-0.1起：**连接客户ERP仅用于读元数据**（对象/字段/事件的结构定义），需要访问白名单强制拦截任何业务表访问
- MVP-0.2起：**允许写元数据**（如自动加字段/校验），仍不触碰业务表
- MVP-2起引入业务数据交互时，优先采用"生成脚本→顾问本地运行"方案（产品仍不直接持有业务数据）
- 顾问身份凭据本地化：金蝶账号、VPN凭据只在本地加密存储
- Agent操作全部可审计：本地记日志，顾问可回溯/回滚

**数据库访问白名单（硬约束）**：
- ✅ 允许：`T_META_*`（元数据表，如T_META_FORM、T_META_FIELD、T_META_OBJECTTYPE）
- ✅ 允许：金蝶官方SDK的MetaData相关接口
- ❌ 禁止：`T_SAL_*`（销售业务表）、`T_BD_*`（基础资料业务表）、`T_STK_*`（库存业务表）、`T_AR_*`（应收业务表）、`T_AP_*`（应付业务表）等
- SQL访问经过"语句审查器"，违规访问直接拒绝并记审计日志
- 即便产品被渗透/提权，白名单层也是硬隔离，无法访问业务数据

### 4.2.1 本地 vs 服务端 数据分配

**必须本地**（安全+用户IP）：

| 数据类型 | 理由 |
|---|---|
| 顾问连接凭据（金蝶账号、VPN、远程凭据） | 安全（MVP-0.1起用于读取BOS元数据） |
| 生成的代码（Python/C#插件） | 顾问/公司IP |
| 操作审计日志 | 合规可追溯 |
| 顾问对话历史原始内容 | 工作隐私 |
| 客户项目CONTEXT.md（含该客户的BOS自定义字段等元数据） | 客户特定知识 |
| 顾问个人偏好memory | 个人数据 |
| 全局配置settings.json | 本地配置 |

**客户业务数据**：**不存任何地方**（产品根本不访问）

**必须服务端**（商业化+共享+算力）：

| 数据类型 | 理由 |
|---|---|
| LLM推理调用 | 算力在云端（走用户自己的API Key，非我们服务器） |
| License验证/账号体系（企业版） | 订阅状态、防盗版 |
| 产品使用遥测（匿名+聚合） | 产品改进+管理者面板 |
| 团队管理者面板数据（企业版） | 老板订阅版功能 |
| 产品版本更新 | OTA升级 |

**GitHub托管 + 本地缓存**（知识+内容）：

| 数据类型 | 理由 |
|---|---|
| 金蝶结构化知识库（JSON） | 全用户共享、统一维护、支持离线 |
| 文档RAG语料+向量索引 | 同上 |
| 插件模板库 | 同上 |
| 业务域Skills（销售/采购/库存/财务/基础资料专家） | 同上 |

### 4.2.2 对话数据流（MVP-0.1）

```
顾问输入需求描述
    ↓
[本地] Agent调用：
  ├─ 内置金蝶标准元数据 + 模板 + 知识RAG（产品自带）
  └─ 客户BOS元数据只读Tool（按需连客户ERP读取客户定制过的字段等结构信息）
    ↓
发送云端LLM（包含：需求描述 + 标准规范 + 该客户的元数据结构）
    ↓
LLM返回代码/方案
    ↓
呈现给顾问（顾问复制粘贴到BOS部署）
```

**关键**：LLM看到的只有**结构信息**（字段叫什么、类型是什么、有哪些事件）——**没有任何业务数据**（客户名、金额、单号都不出现）。元数据是开发规范，即便传云端也无商业机密泄露风险。

### 4.2.3 脱敏层（MVP-2+ 按需引入）

当MVP-2引入"业务API自动执行"能力时，Agent可能会临时需要业务数据上下文。届时在Agent → LLM请求前插入脱敏层。

**替代方案（更干净）**：MVP-2的业务操作可以设计成"Agent生成脚本→顾问本地运行"，产品完全不持有业务数据，脱敏层也不必要。本项目倾向此方案。

### 4.4 分发与更新（用户零依赖）

**用户视角**：
1. 下载 `OpenDeploy-Setup.exe`（含初始知识库，开箱即用）
2. 安装启动
3. 产品后台HTTPS拉取知识库更新
4. **用户无需安装git、Node.js、任何依赖**

**技术实现**：走 jsDelivr CDN + GitHub API + gitee镜像 三层fallback，纯HTTPS拉取不依赖git可执行文件。

### 4.5 版本适配策略（AI自适应为主）

基于"**真实元数据 = 真相来源**"原则，大部分版本差异靠运行时元数据自适应：

| 差异类型 | 能靠AI+元数据自适应吗 | 占比 |
|---|---|---|
| 字段/对象结构差异 | ✅ | ~85% |
| 客户定制字段 | ✅ | — |
| 插件代码模板语法 | ❌（需要固化） | ~10% |
| API参数签名 | ❌（需要固化） | ~5% |

**知识库扁平化，不做version子目录**。MVP-0.1 以 V9.1 为基准，其他版本靠AI+元数据自适应。

### 4.6 多语言（i18n）架构

**支持范围**：主语言 zh-CN，次要 en-US。

- **UI层**（react-i18next）：locales/zh-CN/, locales/en-US/
- **知识库双语结构**：JSON字段携带双语值（`displayName: { "zh-CN": ..., "en-US": ... }`）
- **Skills/docs**：按语言分目录
- **AI对话**：自动识别用户语言并用对应语言回复
- **GitHub文档**：README.md（英）+ README.zh-CN.md（中）

### 4.7 技术栈

| 层 | 选型 | 理由 |
|---|---|---|
| 桌面框架 | Electron | 生态成熟，Cline/Cursor参考多 |
| 前端 | TypeScript + React | Claude编程最顺 |
| UI | Radix UI / shadcn/ui + Tailwind CSS | 现代设计、定制自由 |

### 5.2 MVP-0.1 详细设计（2-3月交付）

#### 目标支持矩阵

| 维度 | MVP-0.1 | 不支持 |
|---|---|---|
| 产品 | 金蝶云星空 | 苍穹、精斗云、KIS |
| 部署模式 | **私有部署** | **SaaS/公有云**（技术做不了） |

| 产品包 | 标准版 + 专业版 | 行业包部分适用 |

#### 范围

1. **Electron桌面应用**（Windows 10/11）
2. **多轮对话UI**（流式输出、代码高亮、Markdown）
3. **客户BOS元数据只读连接**（关键能力）
   - 顾问为每个客户项目配置连接（SDK Token 或 DB连接串）
   - 元数据读取：业务对象/字段/事件/API规范
   - 白名单硬拦截业务表
   - 本地缓存到 `projects/<客户>/metadata.json`
4. **金蝶通用知识库**（混合方案，扁平化）
   - 结构化工具查询为主
   - 文档RAG兜底
   - 5模块覆盖：**销售/采购/基础资料**（深度≥80%场景）+ **库存/财务**（核心50-60%场景，避开结转/关账/成本深水区）
   - 从GitHub HTTPS静默更新
5. **Agent Loop**（需求澄清 + 方案生成 + 代码生成）
6. **Python插件代码生成器**（针对星空BOS Python插件规范）
7. **多LLM provider支持**（Claude / GPT / DeepSeek / Qwen / Ollama 自由切换）
8. **代码展示 + 复制按钮**（顾问手动部署到BOS）
9. **中英双语支持**（i18n）
   - UI：中文 + 英文，可切换
   - AI对话：自动识别用户语言并用相同语言回复
   - 知识库JSON字段双语（zh-CN + en-US）
   - Skills/docs目录按语言分
   - 代码注释语言跟随用户设置
   - GitHub README双语 (`README.md` + `README.zh-CN.md`)

#### 不做什么（硬边界）

- ❌ BOS写入/自动化（加字段、建单据、改配置等，MVP-0.2再做）
- ❌ C# DLL编译部署（MVP-0.2）

> - 超限时仅提示还是要人工审批放行？
>
> AI生成Python插件代码 + 部署指引。顾问复制贴到BOS → 测试通过 → 交付客户。
